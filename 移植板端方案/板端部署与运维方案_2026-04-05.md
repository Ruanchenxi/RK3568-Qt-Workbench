## 2. 基础事实对齐

### 2.1 开发板上不是 .exe

RK3588 Ubuntu 上的可执行程序是 **Linux ELF 二进制**，没有扩展名。

当前编译出来的 `/home/linaro/dev/build-rk3568/RK3568` 就是最终形态。

### 2.2 打包形态对比

| 形态 | 适用场景 | 优缺点 |
|------|---------|-------|
| `.deb` 包 | 正式产品交付 | 规范、可卸载、可追版本；需要写 control 文件 |
| `.tar.gz` + install.sh | 内部联调过渡 | 简单，Qt 库一起打包，解压即用 |
| 直接拷贝二进制 | 开发期 | 最快但容易漏 Qt 依赖 |

**建议**：短期用 `tar.gz + install.sh`，后期转 `.deb`。

### 2.3 打包内容清单

打包时必须带上的东西：

- RK3568 主程序 ELF
- Qt 运行时库（除非板端已全量安装 Qt5）
- 启动包装脚本 `start-rk3568.sh`（设置 DISPLAY、XAUTHORITY、环境变量）
- 默认配置文件 `settings.ini`
- systemd 服务文件 `rk3568.service`
- 板端 start.sh（后端服务恢复脚本）
- 版本信息文件 `VERSION`（包含 git hash 和编译时间）

## 3. 开机自启动方案

### 3.1 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **systemd user service** | 规范、崩溃自动重启、日志可追 | 需要 linger、依赖 X11 就绪 |
| **~/.config/autostart/*.desktop** | 最简单，桌面登录后自动起 | 崩溃不重启，依赖桌面会话 |
| **LXDE/XFCE autostart** | 和桌面环境绑定 | 可移植性差 |
| **rc.local** | 传统方式 | 跑得太早，X11 还没起，不适合 |

**推荐方案**：`systemd user service + autostart desktop` 双保险
- systemd user service 负责崩溃恢复
- desktop autostart 确保桌面起来后第一时间启动

### 3.2 启动包装脚本

文件：`/home/linaro/rk3568/bin/start-rk3568.sh`

```bash
#!/bin/bash
export DISPLAY=:0
export XAUTHORITY=/home/linaro/.Xauthority
export QT_QPA_PLATFORM=xcb
# 如后续确认 XI2 touch 有稳定复现问题再加 QT_XCB_NO_XI2=1

# 等 X11 就绪（最多 30 秒）
for i in {1..30}; do
    xset -q >/dev/null 2>&1 && break
    sleep 1
done

cd /home/linaro/rk3568
exec ./bin/RK3568
```

### 3.3 systemd user service

文件：`~/.config/systemd/user/rk3568.service`

```ini
[Unit]
Description=RK3568 Power Control App
After=graphical-session.target

[Service]
ExecStart=/home/linaro/rk3568/bin/start-rk3568.sh
Restart=on-failure
RestartSec=3
StartLimitIntervalSec=60
StartLimitBurst=5
StandardOutput=append:/home/linaro/rk3568/logs/stdout.log
StandardError=append:/home/linaro/rk3568/logs/stderr.log

[Install]
WantedBy=default.target
```

**关键参数说明**：
- `Restart=on-failure`：非零退出才重启，正常退出不重启
- `RestartSec=3`：崩溃后等 3 秒再拉起，避免快速失败循环
- `StartLimitBurst=5` + `StartLimitIntervalSec=60`：60 秒内最多重启 5 次，超过就停止自动拉起（保护用户免受闪屏地狱）

### 3.4 启用命令

```bash
loginctl enable-linger linaro
systemctl --user enable rk3568.service
systemctl --user start rk3568.service
```

## 4. 日志目录设计

### 4.1 目录结构

```
/home/linaro/rk3568/
├── bin/
│   ├── RK3568                      # 主程序
│   └── start-rk3568.sh             # 启动脚本
├── config/
│   └── settings.ini                # 主配置
├── logs/
│   ├── app/                        # Qt 程序业务日志
│   │   ├── app-2026-04-05.log      # 按天滚动
│   │   └── app-2026-04-04.log
│   ├── serial/                     # 串口协议日志
│   │   └── serial-2026-04-05.log
│   ├── http/                       # HTTP 客户端/服务端日志
│   │   └── http-2026-04-05.log
│   ├── crash/                      # 崩溃栈
│   │   └── crash-2026-04-05-143022.log
│   ├── stdout.log                  # systemd 重定向
│   ├── stderr.log                  # systemd 重定向
│   └── ticket_store.json           # 票据持久化
├── backup/
│   └── ticket_store.20260404.json  # 票据历史备份
└── VERSION
```

### 4.2 日志分级与保留策略

| 日志类型 | 保留时间 | 滚动策略 | 理由 |
|---------|---------|---------|------|
| 业务日志 (app) | **30 天** | 按天分文件，单文件上限 50MB | 排障够用，不占板子 |
| 串口日志 (serial) | **14 天** | 按天 | 协议级细节，短期留存 |
| HTTP 日志 | **14 天** | 按天 | 和工作台联调排障 |
| 崩溃栈 (crash) | **90 天** | 每次崩溃一个文件 | 罕见但必须长期保留 |
| stdout/stderr | **7 天** | systemd logrotate | 最后兜底 |
| **总磁盘预算** | — | **1GB 上限** | 板端存储有限 |

### 4.3 清理实现策略

**建议**：程序启动时检查 `logs/` 目录总大小，超限从最旧的 app 日志开始删。原因：

- 不依赖 cron，自包含
- 部署简单
- 崩溃/异常下也能保证空间

## 5. 票据保存与清理策略

### 5.1 当前 ticket_store.json 存储内容

- 进行中的票（transferState=sending/success）
- 等待回传的票（returnState=pending/uploading）
- 等待删除验证的票（returnState=delete-verifying）
- 撤销中的票（cancelState=accepted/pending）

### 5.2 分级保留策略

| 票据状态 | 保留时长 | 清理时机 | 理由 |
|---------|---------|---------|------|
| 进行中/未完成 | **永久** | 只有人工取消才删 | 丢了就是业务事故 |
| 已回传成功 + 钥匙已清理 | **7 天** | 启动时扫描 | 留一段时间便于现场复查 |
| 已撤销完成 | **7 天** | 启动时扫描 | 同上 |
| 失败/异常终态 | **30 天** | 启动时扫描 | 排障需要 |

### 5.3 清理触发时机

**推荐**：启动时扫描 + 每天零点扫描，两个都做。

代码量很少，鲁棒性明显提升。实现建议：
- 启动初期，在 `TicketStore::loadFromDisk()` 之后执行一次清理
- 运行中，用 `QTimer` 每 24 小时触发一次

### 5.4 清理前必做的备份

**在删除前先备份到 `backup/ticket_store.YYYYMMDD.json`**，备份保留 90 天。

原因：如果某次清理逻辑写错了把活跃票删了，还能回捞。这类逻辑一旦出 bug 就是灾难。

## 6. 产品经理视角补充项

| 问题 | 说明 | 建议 |
|------|------|------|
| **时间同步** | 板子断电后 RTC 可能漂移，票据时间戳错了就是事故 | 开机时检查时间，异常时用 NTP 或提示用户 |
| **程序版本显示** | 现场出问题需要知道"当前跑的是哪个版本" | 主界面某个角落显示版本号 + 编译时间 |
| **网络断开容忍** | 后台 API 连不上时的用户提示要明确 | 登录页已有"服务启动中"，要确认不是无限转圈 |
| **硬件故障提示** | 串口拔了、触摸屏坏了，用户要能看出来 | 系统设置页或状态栏显示硬件连接状态 |
| **异常重启** | systemd 反复重启程序，用户看到闪屏 | 连续崩溃 N 次进入"安全模式"，显示错误信息 |
| **设备唯一标识** | 多个板子部署后如何区分？ | 配置里写 `device/stationId`，启动时打印到日志 |
| **数据导出** | 现场排障时如何把日志拿走？ | 日志目录可直接 U 盘拷，或系统页提供"一键打包日志"按钮 |

## 7. 用户视角补充项

| 场景 | 用户真实心理 | 需要做的 |
|------|------------|----------|
| 开机后等程序起来 | "怎么这么慢，死机了？" | 启动画面/进度条，明确告诉用户现在在干什么 |

## 8. 开发者视角补充项

| 项 | 说明 |
|----|------|
| **远程更新机制** | 每次更新都带 U 盘过去吗？考虑 SSH 推送 + 检查脚本 |

1. systemd user service + 启动脚本 + 开机自启

5. 完整的票据保留/清理策略（7/30/90 天）

7. 日志总量上限保护（1GB）

9. 一键导出日志包
10. 远程更新脚本
11. schema 版本迁移

## 10. 需要现场确认的三件事

落地前必须先确认：

1. **板子上的 Ubuntu 桌面环境是什么？**
   - LXDE / XFCE / GNOME / 裸 X11？

2. **一个板子只服务一个变电站，还是多个？**
   - 影响 stationId 配置

3. **现场有没有网络？**
   - 有网络：远程更新、日志回传、NTP 时间同步都有基础
   - 无网络：一切必须本地自包含

## 12. 偶发异常的防御性改动（2026-04-05 已落地）

针对 2026-03-31 曾经出现过一次的"板端触摸失效"偶发事件（不可复现），已在代码层完成两项**非侵入**改动，定位是"留证据 + 降影响"，不对现有事件链路做任何破坏性改动：

### 12.1 MainWindow 触摸事件诊断钩子

- 文件：`RK3568/src/app/mainwindow.{h,cpp}`
- 动作：主窗口 `event()` 只观察 `TouchBegin/Update/End/Cancel`，经 `LOG_DEBUG("input-touch", ...)` 记录后照常派发。
- 作用：下次偶发时可一眼判定"触摸有没有进到 Qt"，以此分流 Qt 侧 vs X11/驱动侧的排障路径。
- 影响面：零交互变化，DEBUG 级别日志不影响主线信息日志。

### 12.2 SystemPage "重启本程序"按钮

- 文件：`RK3568/ui/systempage.ui`、`RK3568/src/features/system/ui/systempage.{h,cpp}`
- 动作：系统设置页顶部工具栏新增红边红字按钮，点击二次确认后 `qApp->exit(1)`。
- 配套：板端 systemd user service 必须配置 `Restart=on-failure`，非零退出码自动拉起，3~5 秒恢复。
- 作用：异常情况下用户自救入口，不依赖远程介入、不需要重启整机。
- 放置位置为系统设置页而非顶部状态栏，避免高频区误触。

### 12.3 本次没有做的事

不做 `QT_XCB_NO_XI2` 全局降级、不做输入看门狗、不做全量 `WA_AcceptTouchEvents`、不重写任何 `mousePressEvent`。单次偶发不足以支撑上述大范围改动。

详情与"下次偶发时的取证动作清单"见：

- `D:\Desktop\QTtest\RK3568\程序卡死问题\2026-04-04_板端触摸输入异常详细记录.md` 第 14 节

## 2026-04-06 补充说明

以下内容是在原方案基础上补充的“正式交付级缺口”，用于把板端部署方案从“能跑”进一步收口到“能长期稳定交付”。

### A. 交付级补充项

#### A.1 单实例保护

当前方案同时讨论了：

- `systemd user service`
- `desktop autostart`

如果两条链同时生效，而程序自身没有单实例保护，就可能出现：

- 程序双开
- 串口竞争
- `ticket_store.json` 同时写入
- 日志互相覆盖
- H5 / 本地状态机错乱

建议：

1. 正式交付态优先只保留一条主启动链。
2. 若保留双保险，必须补单实例锁：
   - `QLockFile`
   - 或启动脚本先检查旧实例
   - 或程序启动时检查同名实例并拒绝第二实例

#### A.2 自动登录前提

若采用：

- `systemd user service`
- `~/.config/autostart/*.desktop`

则必须明确一个前提：

- **板端桌面用户必须自动登录**

否则图形会话没起来时：

- `DISPLAY=:0` 不可用
- X11 尚未就绪
- GUI 程序不会正常显示

建议在部署步骤中明确写入：

1. 启用 `linaro` 图形桌面自动登录
2. 再启用 RK3568 自启动服务

#### A.3 正式部署目录分层

当前文档使用的是：

- `/home/linaro/rk3568/`

这个路径适合联调和过渡，不够像正式交付形态。

更推荐的正式目录分层：

- 程序：`/opt/rk3568/`
- 配置：`/etc/rk3568/`
- 运行态数据：`/var/lib/rk3568/`
- 日志：`/var/log/rk3568/`
- 备份：`/var/lib/rk3568/backup/`

短期可以先不迁移，但文档里应明确：

- `/home/linaro/rk3568/` 是过渡路径
- 正式量产版建议迁到系统级目录

#### A.4 `ticket_store.json` 必须原子写入

`ticket_store.json` 已经是板端恢复的关键文件。

如果写文件时：

- 断电
- 程序崩溃
- 板子死机

就可能把 JSON 写坏，造成：

- 活跃票丢失
- 程序无法恢复
- 状态机失真

建议采用原子写策略：

1. 先写入 `ticket_store.json.tmp`
2. `flush/fsync`
3. 再 `rename` 覆盖正式文件

并且：

- 读取失败时优先尝试最近一次备份
- 恢复失败要给出明确错误提示，而不是静默清空

#### A.5 运行态状态与审计留痕分离

当前文档重点已经覆盖：

- 调试日志
- 运行恢复

但正式交付后，还需要一类更稳定的业务留痕，回答：

- 谁登录了
- 哪张票什么时候发起
- 是第一次成功还是重试后成功
- 回传是否成功
- 删除验证何时完成

因此建议把两类数据分开：

1. 运行态状态
   - 用于恢复和继续执行，例如 `ticket_store.json`
2. 审计留痕
   - 用于复盘和追责，例如 `audit/`、`business/` 或 `events-YYYY-MM-DD.log`

这样可以避免调试日志被清理后，业务关键事实也跟着丢掉。

#### A.6 禁用屏保、锁屏、休眠

板端现场最常见的“误以为系统坏了”的问题之一就是：

- 屏保
- 自动锁屏
- 自动休眠
- DPMS 熄屏

建议直接升为 P0：

1. 禁用屏保
2. 禁用自动锁屏
3. 禁用自动休眠
4. 禁用 DPMS 黑屏

原则：

- 工控板交付态默认长亮
- 不让用户把黑屏误判为程序卡死

#### A.7 串口权限与设备权限

正式板端部署不能默认假设：

- 程序运行用户一定有权限访问读卡串口和钥匙串口

必须补充：

1. 程序运行用户是谁
2. 是否属于 `dialout` 等串口权限组
3. 是否需要 `udev` 规则
4. 启动时若串口权限不足，应在日志和界面上明确可见

#### A.8 配置与票据 schema 迁移

当前已经提到了：

- `schemaVersion`
- 配置迁移

建议再明确一条：

- 升级前备份旧配置和旧 `ticket_store.json`
- 新版本迁移失败时支持回退到旧版本可识别状态

否则后续版本升级很容易因为本地 schema 不兼容而把现场状态搞乱。

### B. 配置权限边界补充

这一节专门回答一个实际交付问题：

- **系统设置页面里的“钥匙串口”和“读卡串口”，现在是不是只让 admin 改？**

#### B.1 当前代码结论

**是的，当前已经做了“仅管理员可修改串口配置”的限制。**

从当前代码来看：

- `RK3568/src/features/system/ui/systempage.cpp`

里面有两层保护：

1. UI 层禁用
   - `updateSerialPermissionState()` 会根据当前用户权限决定：
     - `keySerialCombo` 是否可用
     - `cardSerialCombo` 是否可用
   - 非管理员时：
     - 下拉框会被禁用
     - cursor 变成普通箭头
     - tooltip 会提示“仅管理员可修改串口配置”

2. 保存时兜底
   - 即使有人绕过 UI，最终走到 `saveSettings()` 时，代码也做了兜底：
   - 如果 `m_canManageSerialPorts == false`
   - 就强制把 `dto.keySerialPort`
     和 `dto.cardSerialPort`
     回退成当前已保存值

也就是说：

- **非管理员即使触发保存，也不会把串口配置写坏。**

#### B.2 当前“管理员”的判定方式

当前串口权限判断复用了：

- `currentUserCanManageSerialPorts()`
- 它又直接复用了：
  - `currentUserCanManageIdentityMedia()`

当前管理员判定大致是：

1. 账号名是 `admin`
2. 或 `role_name == administrator`
3. 或 `role_name` 包含：
   - `admin`
   - `管理员`

所以更准确地说，当前不是只认一个固定账号，而是：

- **认 admin 类账号或管理员角色**

#### B.3 还需要补清楚的边界

虽然串口配置本身已经有限制，但这不等于“系统设置页所有配置都只允许管理员改”。

当前更准确地说是：

- **串口配置只允许管理员改**
- 其他基础配置项是否允许普通用户修改，需要继续按产品规则单独明确

建议在正式部署前，把配置分级写清楚：

- 普通用户可改：
  - 是否允许改首页地址
  - 是否允许改接口地址
  - 是否允许改 tenantCode
  - 是否允许改站号
- 仅管理员可改：
  - 钥匙串口
  - 读卡串口
  - 恢复/清理类操作
  - 维护和导出类操作

### C. 用户、产品、开发三个视角的进一步补充

#### C.1 用户视角

用户最在乎的不是部署术语，而是：

1. 开机后系统能不能自己起来
2. 黑屏时是不是程序坏了
3. 断电重启后票还在不在
4. 读卡器、钥匙、网络到底谁坏了
5. 出问题时有没有简单恢复手段

因此建议在板端交付态做到：

- 开机直达系统
- 长亮不锁屏
- 明确显示设备状态
- 有“重启本程序”和“导出日志包”这类低门槛自救入口

#### C.2 产品经理视角

产品真正要先定的是：

1. 哪些数据属于运行态
2. 哪些数据属于审计态
3. 完成票、失败票、撤销票分别留多久
4. 是否允许普通用户改基础配置
5. 多板、多站点时如何识别设备身份

也就是说：

- **不是先想程序怎么存，而是先想业务要求怎么留。**

#### C.3 开发程序员视角

开发实现时最应该先补上的，是这些“出一次事故就会很难救”的点：

1. 单实例锁
2. 原子写票据文件
3. 自动登录前提
4. 串口权限检查
5. 版本号与 git hash 可见
6. 升级前备份
7. 升级失败可回滚

这几项的价值，远高于界面再多优化一点点。
