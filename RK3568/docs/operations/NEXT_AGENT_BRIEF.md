## 2. 当前已完成到哪里

### 2.1 输入源

工作台是 `QWebEngine` 黑盒页面，主程序通过本地 HTTP 接收工作台发出的传票 JSON。

当前接收链：

```text
Workbench(QWebEngine)
  -> 本地 HTTP POST
    -> TicketIngressService
      -> TicketStore
        -> KeyManageController
          -> KeyManagePage
```

### 2.2 页面展示

钥匙管理页已经支持：

1. `HTTP服务端报文` 显示收到的 JSON  
2. `系统票数据` 显示票列表  
3. 左侧“当前选中系统票”摘要联动  
4. `传票到钥匙` 按钮走正式发送主链  

### 2.3 协议链

已移植进主程序的纯逻辑：

1. `JSON -> payload`
2. `payload -> frame(s)`
3. 单帧 / 多帧
4. `5A 83 / 5A 03` ACK 驱动续发

### 2.4 回传链

当前主程序已落地单帧回传主链：

```text
Q_TASK(status=0x02)
  -> I_TASK_LOG(0x05)
  -> ACK(5A 05 00 00)
  -> UP_TASK_LOG(0x15)
  -> HTTP /finish-step-batch
  -> DEL
```

补充：

1. 多帧 `UP_TASK_LOG` 代码已实现并通过 replay 验证。  
2. 真实钥匙多帧回传样本仍未拿到。  

### 2.5 初始化 / 下载 RFID

当前主程序已落地两条手工链：

1. `初始化(INIT 0x02)`：后端取 `package-data` -> payload 编码 -> 多帧串口下发 -> ACK 闭环  
2. `下载 RFID(DN_RFID 0x1A)`：后端取 `rfid-data` -> payload 编码 -> 串口下发 -> ACK 闭环  

当前结论：

1. 这两条链当前是**手工触发**，不是 ready 后默认自动前置链  
2. 接入它们后，本地程序的自动回传零回归已通过  

### 2.6 自动传票

自动传票逻辑已预埋，当前默认值：

- `ticket/autoTransferEnabled=true`

但它并不是“收到 JSON 就必发”，仍然受以下状态共同约束：

1. 串口已连接
2. 钥匙在位
3. 钥匙稳定
4. `SET_COM` 握手已完成（`sessionReady=true`）

## 3. 当前已经拍板的业务规则

### 3.1 系统票来源

系统票数据是：

> 工作台/服务器任务的本地映射缓存

不是用户本地临时草稿。  

### 3.2 删除系统票

当前阶段**不做“删除系统票”按钮**。  

原因：

1. 系统票真值来源是工作台/服务器
2. 撤销/删除规则还没定完
3. 先把传票/回传主链做稳，再统一设计撤销生命周期

补充：

1. 长期目标不是“用户手工删除系统票”
2. 正确方向是“工作台撤销事件驱动系统票状态变化”
3. 规则已定：
   - 未传钥匙：可直接移除
   - 已传钥匙：先标记为“已撤销，待清理钥匙任务”，清理后再移除
4. 但当前阶段暂不实现，避免在回传主链未完成前把状态机复杂化

### 3.3 同一任务重复触发“钥匙传票”

如果同一个 `taskId` 已经是：

1. `sending`
2. `success`

再次从工作台触发“钥匙传票”时，当前主程序默认**不重复下发**。

### 3.4 获取系统票列表按钮

当前 `获取系统票列表` 按钮只是：

> 刷新显示本地主程序内已经入池的系统票

它不是：

1. 主动请求工作台/服务器
2. 重新同步服务端票数据

## 8. RK3588 差底座专项说明

1. 当前 RK3588 / Ubuntu / 差底座环境下的核心问题，不是“协议完全不通”，而是：
   - `waitForBytesWritten()` 频繁超时
   - 旧 timeout / 旧回包 / 差接触条件下的重复回包会污染当前链路
   - 板端更像“链路可通但时序很脏”，而不是简单的“完全连不上”
2. 当前主程序已开始按“差底座适配”收口：
   - 顶部设备状态拆成 `待确认 / 恢复中 / 已就绪`
   - stale timeout 防护已加入
   - 重复 `Q_TASK / I_TASK_LOG` 回包已开始忽略
   - 若 `Q_TASK` 发现钥匙中无任务，但系统票卡在 `sending/failed`，会自动回退为 `auto-pending` 并补传
   - 若 HTTP 回传成功，但钥匙中仍残留 `status=0x02` 任务，会自动继续 `DEL`
3. 当前仍待继续分析 / 收口的点：
   - Linux / `/dev/ttyS4` 环境下 `waitForBytesWritten()` 超时与真实有效响应并存的原因
   - 为什么某些成功链路后，后续旧 timeout 仍可能影响显示与业务收口
   - 板端坏链路场景下，是否还需要进一步的“恢复中 -> 自动 SET_COM -> 自动 Q_TASK -> 自动续业务”完整恢复态
4. 对 PRO 做全面分析时，建议重点看：
   - 协议成功判据是否应“响应优先于 write/wait 超时”
   - 现有状态机是否还存在旧包/旧超时污染新命令的问题
   - 是否需要把当前 Linux 板端串口写等待从“失败判据”降级为“诊断告警”
