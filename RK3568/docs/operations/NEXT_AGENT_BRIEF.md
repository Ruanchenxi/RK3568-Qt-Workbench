## 2. 当前已完成到哪里

### 2.1 输入源

工作台是 `QWebEngine` 黑盒页面，主程序通过本地 HTTP 接收工作台发出的传票 JSON。

当前接收链：

```text
Workbench(QWebEngine)
  -> 本地 HTTP POST
    -> TicketIngressService
      -> TicketStore
        -> KeyManageController
          -> KeyManagePage
```

### 2.2 页面展示

钥匙管理页已经支持：

1. `HTTP服务端报文` 显示收到的 JSON  
2. `系统票数据` 显示票列表  
3. 左侧“当前选中系统票”摘要联动  
4. `传票到钥匙` 按钮走正式发送主链  

### 2.3 协议链

已移植进主程序的纯逻辑：

1. `JSON -> payload`
2. `payload -> frame(s)`
3. 单帧 / 多帧
4. `5A 83 / 5A 03` ACK 驱动续发

### 2.4 自动传票

自动传票逻辑已预埋，当前默认值：

- `ticket/autoTransferEnabled=true`

但它并不是“收到 JSON 就必发”，仍然受以下状态共同约束：

1. 串口已连接
2. 钥匙在位
3. 钥匙稳定
4. `SET_COM` 握手已完成（`sessionReady=true`）

## 3. 当前已经拍板的业务规则

### 3.1 系统票来源

系统票数据是：

> 工作台/服务器任务的本地映射缓存

不是用户本地临时草稿。  

### 3.2 删除系统票

当前阶段**不做“删除系统票”按钮**。  

原因：

1. 系统票真值来源是工作台/服务器
2. 撤销/删除规则还没定完
3. 先把传票/回传主链做稳，再统一设计撤销生命周期

### 3.3 同一任务重复触发“钥匙传票”

如果同一个 `taskId` 已经是：

1. `sending`
2. `success`

再次从工作台触发“钥匙传票”时，当前主程序默认**不重复下发**。

### 3.4 获取系统票列表按钮

当前 `获取系统票列表` 按钮只是：

> 刷新显示本地主程序内已经入池的系统票

它不是：

1. 主动请求工作台/服务器
2. 重新同步服务端票数据

### 已知现状

1. 手动传票真机已经开始联调
2. 自动传票真机开始验证

## 5. 下一步应该干什么

继续做 **回传链路**。

建议顺序：

1. 先确认自动传票在真机上的行为是否稳定
2. 再把“回传（重试）”正式接入主程序主链
3. 再继续收 `初始化 / 下载 RFID / 回传` 这几条业务主链

1. 不要继续补测试工具 UI
2. 不要先做系统票删除
3. 不要大规模返工状态机
