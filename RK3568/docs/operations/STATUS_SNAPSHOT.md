## 8. 联调补充结论（2026-03-09）

1. 当前 Windows 本地联调环境下，`localhost` 就是当前主程序所连的有效后端，不需要强制改成 `192.168.3.131`。  
2. 若浏览器访问 `http://localhost/api/kids-outage/third-api/finish-step-batch` 返回 `405`：
   - 说明接口存在
   - 只是浏览器发的是 `GET`
   - 主程序发 `POST` 时可正常工作
3. 对 414 锁出现的回传失败，经确认是：
   - 当前联调用后端缺少 414 的 RFID 数据
   - 后端无法将回传 `RFID` 匹配到闭锁装置
   - 不属于当前主程序回传 HTTP/串口解析主链错误
4. 原产品对同一把 411 锁的单步停电 / 单步送电样本也表现出：
   - 工作台步骤 `rfid` 固定为 `C9:04:99:A0:02`
   - 回传日志 `rfid` 分别出现 `C90499A001` / `C90499A000`
   - 因此当前主程序出现此类现象时，应优先视为“兼容原产品行为”
5. 钥匙管理页中的顶部 `Q_TASK诊断` 与表格旁 `读取钥匙票列表` 当前底层都走 `Q_TASK`，只是一个偏诊断入口、一个偏业务入口
6. `刷新本地系统票列表` 当前仅刷新主程序本地票池，不会重新向工作台/后端主动拉票
7. `transferState` 表示“主程序对传票下发状态的本地判断”，不是钥匙侧真值；当本地状态和钥匙真实任务状态冲突时，当前以 `Q_TASK` 结果做对账修正
8. 顶部“通讯”状态不再只看 `SET_COM ACK`；当前只有在握手完成后又拿到一次真实业务响应（例如 `Q_TASK / I_TASK_LOG / UP_TASK_LOG`）才视为“钥匙已就绪”
