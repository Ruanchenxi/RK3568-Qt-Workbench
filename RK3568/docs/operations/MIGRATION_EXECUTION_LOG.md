## 2. 为什么真机前仍要先做 SET_COM / 初始化

当前这里的“初始化”优先指**现有底座通讯初始化**（串口连接 + 握手 + ready），不是手工 `INIT(0x02)` 业务链。  

原因不是“传票协议本身依赖 SET_COM 字段”，而是：

1. 主程序中的钥匙业务发送，都建立在现有钥匙会话状态机之上。  
2. 当前状态机要求：
   - 串口连接成功
   - `SET_COM` 握手成功
   - 钥匙在位
   - 钥匙稳定
   - 会话 ready
   才允许进入正式业务发送。  
3. 因此回公司后做真机测试时，必须先确认旧链路正常，再测传票。  

简化理解：

> 传票不是绕开钥匙链路的新通道，而是建立在现有钥匙会话之上的新增业务路径。

## 5. 当前产品决策（2026-03-09）

### 5.1 系统票数据的“删除/撤销”规则

当前先不实现“删除系统票数据”按钮，也不在这一阶段处理“工作台撤销后系统票如何自动删除/保留”的完整业务规则。  

原因：

1. 系统票的真值来源是工作台/服务器，不是本地主程序。  
2. 在自动传票主链完全走通前，过早加入“删除系统票”会引入额外状态复杂度。  
3. 更合理的做法是：先把“工作台 JSON -> 系统票入池 -> 手动/自动传票”主链做稳，再统一设计撤销/删除规则。  

当前决策：

1. **暂不增加“删除系统票”按钮**  
2. **撤销后的系统票生命周期，作为后续单独事项设计与实现**  

### 5.2 当前优先级

当前优先做：

1. 自动传票主链闭环
2. 回传链路接入
3. 送电样本与多帧回传样本继续完善
4. 真机验证

撤销/删除规则延后：

1. 工作台撤销后系统票是否自动移除
2. 已传进钥匙的票在撤销后如何标记
3. 是否需要“删除系统票”或“清理已撤销系统票”按钮

### 5.3 重复触发同一任务的当前规则

当同一个 `taskId` 的系统票已经处于以下状态时：

1. `sending`
2. `success`

再次从工作台触发“钥匙传票”时，当前主程序默认**不重复下发**。  

当前处理策略：

1. 若状态为 `sending`：提示“该系统票正在传票中，请稍后再试”  
2. 若状态为 `success`：提示“该系统票已传票到钥匙，本次不重复下发”  

设计原因：

1. 避免同一任务在钥匙中重复下发  
2. 避免在自动传票阶段引入重复状态机  
3. 先把主链稳定，再单独设计“重发/覆盖传票”的业务规则  

补充规则（2026-03-09）：

1. 如果该系统票已经成功传票到钥匙，但随后**钥匙中的对应任务被手动删除（DEL 成功）**
2. 主程序会把该系统票状态改成：
   - `key-cleared`
3. `key-cleared` 的含义是：
   - 钥匙侧同任务已清空
   - 当前系统票允许再次传票
   - 再次从工作台触发“钥匙传票”时，允许重新进入发送链
4. 这个规则只针对“钥匙侧任务已被删除”的场景，不等于通用重复下发放开  

### 5.4 自动传票前置条件（已落地）

自动传票不会只看“钥匙已插”。  
当前自动传票进入发送前，要求以下条件同时满足：

1. 串口已连接  
2. 钥匙在位  
3. 钥匙稳定  
4. `SET_COM` 握手已完成（`sessionReady=true`）  

如果只满足前 3 条但会话未 ready，则系统票应保持在：

- `等待自动传票`

而不是误进入：

- `发送失败`

### 5.5 “获取系统票列表”按钮当前语义

当前实现中，`获取系统票列表` 按钮**不是**重新向工作台/服务器拉数据。  

它当前只做：

1. 刷新并重新显示**本地主程序内已经接收到并入池的系统票数据**  

也就是说：

1. 工作台申请任务 -> JSON 进入本地 HTTP 接收 -> 系统票入池  
2. `获取系统票列表` 只是把当前系统票池重新刷到表格  

它不是：

1. 主动发 HTTP 请求去工作台查票  
2. 从服务器重新同步系统票数据

### 5.7 回传真机联调补充结论（2026-03-10）

1. 当前 Windows 本地联调环境下，`localhost` 就是有效后端，不需要强制切到 `192.168.3.131`
2. 浏览器访问 `http://localhost/api/kids-outage/third-api/finish-step-batch` 返回 `405`
   - 只说明浏览器发的是 `GET`
   - 不代表接口不存在
   - 主程序发 `POST` 时可正常工作
3. 当前主程序发送的回传 HTTP JSON 已被本地后端成功接收并返回 `200`，说明：
   - HTTP 客户端主链可用
   - `opTime=yyyy-MM-dd HH:mm:ss` 的完整时间串可被当前后端接受
4. 对 414 锁出现的回传失败：
   - 后端返回 `400`
   - 提示“未找到步骤序列号为0的RFID对应的闭锁装置”
   - 经确认，当前联调用后端与原产品后端不是同一套，且该后端缺少 414 的 RFID 数据
5. 因此 414 这类失败当前应判定为：
   - **后端业务数据缺失 / RFID 映射未配置**
   - **不是当前主程序回传链路实现错误**
6. 当前联调判断标准应改为：
   - 先用后端已配置 RFID 的锁验证回传主链
   - 对后端未配置 RFID 的锁，不拿来否定主程序实现
7. 原产品与当前主程序的对照进一步确认：
   - 工作台步骤 `rfid` 与回传日志 `rfid` 不是同一语义
   - 单步停电 / 送电回传 `rfid` 可能不同
   - 这类现象当前应优先按“兼容原产品行为”处理，而不是直接视为代码 bug

### 5.9 工作台撤销后的系统票生命周期（已拍板，暂缓实现）

当前已经明确产品规则，但**暂不立刻实现**：

1. 如果工作台撤销了一条**尚未传进钥匙**的系统票：
   - 主程序收到撤销事件后
   - 该系统票可直接从系统票列表移除

2. 如果工作台撤销了一条**已经传进钥匙**的系统票：
   - 主程序不应直接把它从系统票列表删除
   - 应先标记为：
     - `已撤销，待清理钥匙任务`
     - 或同等语义状态
   - 等钥匙侧任务清理完成后再移除

3. 当前推荐的技术实现方式：
   - 不是手工增加“删除系统票”按钮
   - 而是由工作台/后端补一个**撤销事件输入**
   - 例如新增本地 HTTP 输入：
     - `POST /ywticket/WebApi/cancelTicket`
   - 主程序通过 `TicketIngressService -> TicketStore -> KeyManageController` 处理

4. 当前阶段为什么不做：
   - 回传链路还没完成
   - 初始化 / 下载 RFID / 回传 主链还没完全收口
   - 现在过早实现撤销规则，容易把状态机复杂度进一步抬高


### 5.10 RK3588 差底座适配补充结论

1. 当前板端问题的主因不是“功能完全未实现”，而是：
   - 差底座接触不稳
   - Linux / `ttyS4` 下 `waitForBytesWritten()` 经常超时
   - 旧 timeout / 旧回包会在坏链路里污染当前命令状态
2. 当前已经开始收口的补丁包括：
   - 顶部状态改成 `待确认 / 恢复中 / 已就绪`
   - stale timeout 防护
   - 忽略重复 `Q_TASK / I_TASK_LOG` 回包
   - 自动回传前先补一次 `SET_COM`，再发 `I_TASK_LOG`
   - HTTP 成功后若钥匙任务仍残留，自动继续 `DEL`
3. 当前仍待真机确认的关键点：
   - 收到有效 `Q_TASK / I_TASK_LOG / UP_TASK_LOG / DEL` 响应后，是否还会被后续旧 timeout 重新打坏
   - 差底座场景下，是否无需用户再次拔插，也能自行恢复 `SET_COM -> Q_TASK -> 回传/传票`
4. 给 PRO 的分析重点：
   - 板端串口写等待的异常语义
   - 协议响应成功与写等待失败同时出现时，状态机应该以谁为准
   - 是否需要把恢复态进一步做成原产品式的完整自恢复链
