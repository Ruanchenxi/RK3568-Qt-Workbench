## 2. 为什么真机前仍要先做 SET_COM / 初始化

原因不是“传票协议本身依赖 SET_COM 字段”，而是：

1. 主程序中的钥匙业务发送，都建立在现有钥匙会话状态机之上。  
2. 当前状态机要求：
   - 串口连接成功
   - `SET_COM` 握手成功
   - 钥匙在位
   - 钥匙稳定
   - 会话 ready
   才允许进入正式业务发送。  
3. 因此回公司后做真机测试时，必须先确认旧链路正常，再测传票。  

简化理解：

> 传票不是绕开钥匙链路的新通道，而是建立在现有钥匙会话之上的新增业务路径。

## 5. 当前产品决策（2026-03-09）

### 5.1 系统票数据的“删除/撤销”规则

当前先不实现“删除系统票数据”按钮，也不在这一阶段处理“工作台撤销后系统票如何自动删除/保留”的完整业务规则。  

原因：

1. 系统票的真值来源是工作台/服务器，不是本地主程序。  
2. 在自动传票主链完全走通前，过早加入“删除系统票”会引入额外状态复杂度。  
3. 更合理的做法是：先把“工作台 JSON -> 系统票入池 -> 手动/自动传票”主链做稳，再统一设计撤销/删除规则。  

当前决策：

1. **暂不增加“删除系统票”按钮**  
2. **撤销后的系统票生命周期，作为后续单独事项设计与实现**  

### 5.2 当前优先级

当前优先做：

1. 自动传票主链闭环
2. 回传链路接入
3. 初始化 / 下载 RFID / 回传 主链继续完善
4. 真机验证

撤销/删除规则延后：

1. 工作台撤销后系统票是否自动移除
2. 已传进钥匙的票在撤销后如何标记
3. 是否需要“删除系统票”或“清理已撤销系统票”按钮

### 5.3 重复触发同一任务的当前规则

当同一个 `taskId` 的系统票已经处于以下状态时：

1. `sending`
2. `success`

再次从工作台触发“钥匙传票”时，当前主程序默认**不重复下发**。  

当前处理策略：

1. 若状态为 `sending`：提示“该系统票正在传票中，请稍后再试”  
2. 若状态为 `success`：提示“该系统票已传票到钥匙，本次不重复下发”  

设计原因：

1. 避免同一任务在钥匙中重复下发  
2. 避免在自动传票阶段引入重复状态机  
3. 先把主链稳定，再单独设计“重发/覆盖传票”的业务规则  

补充规则（2026-03-09）：

1. 如果该系统票已经成功传票到钥匙，但随后**钥匙中的对应任务被手动删除（DEL 成功）**
2. 主程序会把该系统票状态改成：
   - `key-cleared`
3. `key-cleared` 的含义是：
   - 钥匙侧同任务已清空
   - 当前系统票允许再次传票
   - 再次从工作台触发“钥匙传票”时，允许重新进入发送链
4. 这个规则只针对“钥匙侧任务已被删除”的场景，不等于通用重复下发放开  

### 5.4 自动传票前置条件（已落地）

自动传票不会只看“钥匙已插”。  
当前自动传票进入发送前，要求以下条件同时满足：

1. 串口已连接  
2. 钥匙在位  
3. 钥匙稳定  
4. `SET_COM` 握手已完成（`sessionReady=true`）  

如果只满足前 3 条但会话未 ready，则系统票应保持在：

- `等待自动传票`

而不是误进入：

- `发送失败`

### 5.5 “获取系统票列表”按钮当前语义

当前实现中，`获取系统票列表` 按钮**不是**重新向工作台/服务器拉数据。  

它当前只做：

1. 刷新并重新显示**本地主程序内已经接收到并入池的系统票数据**  

也就是说：

1. 工作台申请任务 -> JSON 进入本地 HTTP 接收 -> 系统票入池  
2. `获取系统票列表` 只是把当前系统票池重新刷到表格  

它不是：

1. 主动发 HTTP 请求去工作台查票  
2. 从服务器重新同步系统票数据

### 5.8 工作台撤销后的系统票生命周期（已拍板，暂缓实现）

当前已经明确产品规则，但**暂不立刻实现**：

1. 如果工作台撤销了一条**尚未传进钥匙**的系统票：
   - 主程序收到撤销事件后
   - 该系统票可直接从系统票列表移除

2. 如果工作台撤销了一条**已经传进钥匙**的系统票：
   - 主程序不应直接把它从系统票列表删除
   - 应先标记为：
     - `已撤销，待清理钥匙任务`
     - 或同等语义状态
   - 等钥匙侧任务清理完成后再移除

3. 当前推荐的技术实现方式：
   - 不是手工增加“删除系统票”按钮
   - 而是由工作台/后端补一个**撤销事件输入**
   - 例如新增本地 HTTP 输入：
     - `POST /ywticket/WebApi/cancelTicket`
   - 主程序通过 `TicketIngressService -> TicketStore -> KeyManageController` 处理

4. 当前阶段为什么不做：
   - 回传链路还没完成
   - 初始化 / 下载 RFID / 回传 主链还没完全收口
   - 现在过早实现撤销规则，容易把状态机复杂度进一步抬高
