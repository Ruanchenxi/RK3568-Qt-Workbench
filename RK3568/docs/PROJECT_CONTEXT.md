## 1. 文档目的

本文件用于让“没有历史上下文的人”快速理解：

1. 这个项目要解决什么问题。  
2. 当前做到什么程度。  
3. 近期开发应优先做什么。  
4. 哪些边界不能随意突破。  

## 2. 项目定义（一句话）

RK3568 项目是对既有可运行产品进行功能复刻与工程重构的项目，目标在 PC 与 RK3588 Ubuntu 双端运行，并通过闭环验证逐步收敛到稳定版本。

## 3. 业务目标与工程目标

### 3.1 业务目标

1. 复刻原产品关键流程，确保用户侧行为一致。  
2. 支持钥匙管理、日志查看、系统配置、工作台业务。  
3. 支持多种登录模式（账号密码、刷卡、后续指纹）。  

### 3.2 工程目标

1. 将平铺代码收敛到可维护目录结构（Feature-First Lite）。  
2. 降低单人维护成本，提升定位速度与回滚安全性。  
3. 建立可执行门禁与无硬件验证路径。  

## 4. 运行平台与部署场景

1. 开发验证平台：Windows（Qt Creator + qmake）。  
2. 目标部署平台：RK3588 Ubuntu。  
3. 构建方式：当前维持 `qmake + .pri`，不在本阶段切 CMake。  

## 5. 页面与能力现状（2026-03-02）

### 5.1 工作台（Workbench）

1. 主要接口链路已可用。  
2. 后续可能随后端 API 调整进行适配。  

### 5.2 服务日志（Log）

1. 主流程已可用。  
2. RK3588 板端当前已明确以原产品 `AdapterService` 为对照口径：  
   - 原产品前端：`/home/linaro/AdapterService/bin/AdapterService`
   - 原产品恢复脚本：`/home/linaro/outage/start.sh`
   - 原产品真实服务日志：`/home/linaro/outage/target/kids/log/info-*.log` 与 `error-*.log`
3. 后续重点不再是“泛化日志源猜测”，而是板端优先对齐原产品日志路径与可读性。  
4. 当前实现上，“服务日志页是否打开”与“是否允许登录”已解耦：
   - 服务日志页负责日志查看与服务侧状态观察
   - 登录门禁以后端实际 ready 探测为准，不以日志页展示状态为准
5. 当前实现上，切换用户不会主动重启后端服务：
   - Linux 默认 `remote` 模式下，通常在首次进入服务日志页时启动日志控制器
   - 非 Linux `remote` 模式下，也可能在日志页构造阶段直接启动
   - 当前主流程更偏向“探测服务可达 + 跟踪日志”，不是重新拉起本地服务脚本  

### 5.3 钥匙管理（Key）

1. 传票主链已并入主程序：工作台 JSON -> 本地 HTTP 接收 -> 系统票池 -> 手动/自动传票。  
2. 串口报文页、HTTP 客户端/服务端报文页已承担主链路诊断作用。  
3. 自动回传主链已在本地程序真机环境下验证通过。  
4. 手工 `初始化(INIT 0x02)` 与手工 `下载 RFID(DN_RFID 0x1A)` 已正式接入主程序。  
5. 当前重点已转向：送电样本、真实多帧回传样本、以及后续业务收口。  

### 5.4 系统设置（System）

1. 基础配置已可用，并已收口为更偏设备产品形态的简化表单。  
2. 基础配置页当前主交互是 `保存并应用`。  
3. 串口参数固定为 `115200 / 8`，不再允许用户编辑；钥匙串口 / 读卡串口走自动识别 + 下拉选择。  
4. `保存并应用` 按钮已同时具备：
   - `.ui` 静态样式可见
   - 运行时再次显式设样式与尺寸
   便于在 Qt Designer 中直接调整。  
5. 用户身份采集模块作为后续刷卡/指纹登录配套能力。  

### 5.5 登录（Auth）

1. 账号密码登录已可用。  
2. 指纹能力仍处于预留插槽阶段，后续逐步接入。  
3. 刷卡登录第一版已按原产品链路接入：
   - 登录页接入读卡串口采集源
   - 默认按 `/dev/ttyS3`、`9600 8-N-1` 打开
   - 成功后沿用密码登录成功的统一 UI 状态刷新链
4. 用户身份采集页当前已进入“真实采卡落库”阶段：
   - 采集卡号复用同一条读卡串口真值
   - 用户列表默认尝试从 `list-user` 解析用户详情
   - 保存卡号默认调用 `update-card-no`
   - 若接口未返回 `userId`，页面仅允许查看，不允许保存卡号
5. 用户身份采集页当前已按原产品验证结果收口权限边界：
   - 刷新用户列表可对普通已登录角色开放
   - `采集卡号 / 删除卡号` 以管理员权限为准
   - 前端默认规则为：非管理员只读，管理员可维护卡号
6. 用户身份采集页当前交互优先保证工业现场安全：
   - 仍保留显式 `采集卡号` 按钮，不采用“选中用户后直接刷卡即自动绑定”的激进模式
   - 若刷到的卡已绑定其他用户，前端会先阻止换绑，并提示先删除原绑定
7. 当前登录页已补服务就绪门禁：
   - 登录页显示时会通过 `list-account` 做轻量探测
   - 当前探测已改为异步轮询，不在 GUI 线程内做同步阻塞等待
   - 服务未就绪时，不开放账号密码登录与刷卡登录
   - 这样更接近原产品“服务启动完成后再允许登录”的行为
8. 当前底部硬件状态栏已开始从“纯占位文案”向真实状态过渡：
   - 读卡器状态已接入应用内读卡源的状态变化
   - 当前可区分：`空闲 / 检测中 / 就绪 / 异常 / 未配置串口`
   - 指纹仪当前仍未接入真实驱动，页面和底部状态栏应诚实显示为“未接入”
9. 当前读卡器状态链路仍属于应用内读卡源复用策略：
   - 登录页与系统设置页复用同一套读卡源口径
   - 底部状态当前跟随这些页面内读卡源的生命周期变化
   - 这一步优先避免串口抢占，尚未升级为独立常驻硬件守护服务

### 5.6 虚拟键盘（Keyboard）

1. 当前正式主线为应用内自定义 QWidget 键盘。
2. 登录页 / 系统设置页为完整支持页。
3. 工作台页按 QWebEngine 特例 bridge 处理，需先聚焦网页可编辑元素。
4. 钥匙管理页 / 服务日志页当前不作为有效输入页。

## 6. 外部依赖与硬件边界

1. 原产品硬件：X5 钥匙底座 + X5 钥匙。  
2. 对照策略：原产品行为对照 -> 新系统闭环复刻。  
3. 串口规划红线：  
   - `/dev/ttyS4`：钥匙协议链路  
   - `/dev/ttyS3`：刷卡采集链路（当前已接入）  
4. 登录能力通过外部 API 对接（账号/刷卡/指纹均可能走 API）。  

## 7. 当前技术约束（必须遵守）

1. `.ui` 文件固定在项目根 `ui/`，不迁移。  
2. 协议关键语义（7E6C/CRC/重试/稳定窗）不随意改。  
3. 页面层不得直连底层实现（串口/网络/进程）。  
4. 每步改动必须满足可编译、可运行、可回滚。  
5. 结构调整后必须执行门禁：`.\tools\arch_guard.ps1 -Phase 3`。  

## 8. 当前开发方法论

1. 小步提交：每次改动聚焦单一问题。  
2. 闭环验证：每个链路（例如 SET_COM/Q_TASK/DEL）单独验收。  
3. 可回滚优先：先保证安全，再追求速度。  
4. 文档同步：规则变更时文档与脚本必须同步更新。  

## 9. 近期优先级（建议顺序）

1. 继续收 **送电任务样本** 与 **真实多帧回传样本**。  
2. 保持 `传票 / 手工 INIT / 手工下载 RFID / 自动回传` 主链稳定。  
3. 在主链完成后，再统一处理撤销/删除系统票、重复下发高级规则等业务 bug。  
4. 保持已完成功能稳定，避免结构性回退。  

## 10. 非目标（当前阶段不做）

1. 不进行大规模命名重构。  
2. 不切换 CMake。  
3. 不在未验证前重写协议核心语义。  

## 11. 新 Agent 最小上手路径（执行版）

1. 读 `docs/README.md`（总入口）。  
2. 读本文件（业务与约束背景）。  
3. 读 `architecture/CURRENT_FOLDER_LAYOUT.md`（当前结构）。  
4. 读 `architecture/CODEMAP.md`（修改入口定位）。  
5. 读 `architecture/DEPENDENCY_RULES.md`（依赖红线）。  
6. 若涉及串口协议改动，必须先读：  
   - `serial/SERIAL_PACKET_AGENT_GUIDE.md`（链路架构、状态机、改动流程）  
   - `serial/PROTOCOL_FRAMES_REFERENCE.md`（逐字节帧格式、CRC 向量、常量速查）  
7. 读 `replay/REPLAY_SPEC.md`（无硬件验证）。  
8. 修改后执行 `.\tools\arch_guard.ps1 -Phase 3` 并完成主路径验证。
