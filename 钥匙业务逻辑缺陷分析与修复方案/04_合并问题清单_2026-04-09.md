## C1：`onDeleteClicked()` 未记录 sessionId，手工删除必定卡死

### 现象
管理员在钥匙管理页选中一条钥匙任务，点"删除"。DEL 命令发出且可能已成功，但页面永远停在"正在对账确认钥匙任务是否已删除"，无法收口。

### 根因
`onDeleteClicked()`（KeyManageController.cpp line 632-661）：
- 记录了 `m_pendingDeletedKeyTaskRaw`、`m_pendingDeletedSystemTicketId`
- **没有记录 `m_pendingDeleteSessionId`**（默认值 0）
- 没有重置 `m_pendingDeleteRetryCount`

后续 `finalizePendingReturnDelete()`（line 1730）：
- `m_pendingDeleteSessionId == 0` → `sessionUnknown = true`
- 进入 line 1753-1758 分支："删除链路曾被钥匙移除打断，继续等待原钥匙回位"
- return 不清理 pending 状态 → 下次 Q_TASK 又走同一分支 → 无限循环

### 涉及代码
- `KeyManageController.cpp`：`onDeleteClicked()`（line 632-661）
- `KeyManageController.cpp`：`finalizePendingReturnDelete()`（line 1730-1798）

## C2：撤销链用当前钥匙数据替代原钥匙判断

### 现象
任务传到钥匙 A → 工人拿走钥匙 A → 底座放上钥匙 B → 工作台发撤销 → 系统拿钥匙 B 的数据做判断 → 可能错误判定"任务已不在钥匙中" → 删除本地票返回"已取消"。但任务实际还在钥匙 A 上。

### 根因
`m_lastKeyTasks` 在每次 Q_TASK 回来时被无条件覆盖为当前钥匙数据（line 1088）。

撤销链两处关键判断都依赖它：
1. cancel ingress handler line 443：`!taskExistsInList(m_lastKeyTasks, taskId)` → 直接删票
2. `tryProcessPendingCancelTasks` line 1976-1984：`findKeyTaskIdRaw` 返回空 → 直接删票

入口保护（line 428-438）不完整：只拦了"无钥匙/无数据"的情况，如果钥匙 B 在位且有任务，保护被绕过。

### 涉及代码
- `KeyManageController.cpp`：cancel ingress lambda（line 334-492）
- `KeyManageController.cpp`：`tryProcessPendingCancelTasks()`（line 1939-2026）
- `KeyManageController.cpp`：`m_lastKeyTasks` 赋值（line 1088）

## C3：手工删除和回传自动清理共用 pending delete 状态机

### 现象
管理员手工删除 和 回传后自动清理 都写入同一组变量（`m_pendingDeletedSystemTicketId` 等），都走 `finalizePendingReturnDelete()` 收口。导致：
- 手工删除被卷入回传链的复杂对账状态
- 回传清理可能被手工删除覆盖 pending 状态

### 根因
`onDeleteClicked()`、`tryAutoCleanupReturnedTicket()`、`markReturnDeletePending()` 三个入口共用同一组 pending delete 变量。

### 涉及代码
- `KeyManageController.cpp`：`onDeleteClicked()`（line 632-661）
- `KeyManageController.cpp`：`tryAutoCleanupReturnedTicket()`（line 2172-2217）
- `KeyManageController.cpp`：`markReturnDeletePending()`（line 1719-1728）
- `KeyManageController.cpp`：`finalizePendingReturnDelete()`（line 1730-1798）

## C4：传票标记 success 太早，Q_TASK 验证失败不回滚

### 现象
传票帧全部发完并收到 ACK → 立刻标 `success` → 后续 Q_TASK 验证如果钥匙里没有这个任务 → success 不回滚。

注意：工作台前端根据 HTTP 接口返回决定是否隐藏"钥匙传票"按钮，即使本地回滚了 success，工作台前端不会感知。此修复主要服务于**管理员查看本地票状态**的准确性。

### 根因
`reconcileSystemTicketsFromKeyTasks()`（line 2093-2119）只做正向纠正（本地状态落后 → 纠正为 success），不做反向纠正（本地 success 但钥匙里没有 → 不处理）。

### 涉及代码
- `KeyManageController.cpp`：`TicketTransferFinished` handler（line 1204-1240）
- `KeyManageController.cpp`：`reconcileSystemTicketsFromKeyTasks()`（line 2093-2119）

## C5：自动清理 DEL 和撤销 DEL 可能竞争同一任务

### 现象
一个任务同时处于 `return-upload-success`（回传已上传成功）和 `cancel-accepted`（撤销已受理），`tryAutoCleanupReturnedTicket` 和 `tryProcessPendingCancelTasks` 可能各自对它发 DEL，竞争记账。

### 根因
`tryAutoCleanupReturnedTicket()`（line 2172-2177）检查了 `m_activeReturnTaskId` 和 `m_pendingDeletedSystemTicketId`，但**没检查 `m_pendingCancelSystemTicketId`**和票据的 `cancelState`。

### 涉及代码
- `KeyManageController.cpp`：`tryAutoCleanupReturnedTicket()`（line 2172-2217）

## C6：孤儿恢复仍会接管外来钥匙任务进本地票池

### 现象
插入外来钥匙，Q_TASK 回来后，钥匙里有但本地没有的任务被 `ingestOrphanTicket` 自动建成本地记录。

### 当前状态
**部分已修复**：孤儿票设置了 `transferState = "orphan-recovered"`、`returnState = "manual-required"`，不会被自动回传/清理。但仍会出现在本地系统票列表中。

cancel ingress handler（line 366-371）对不存在的 taskId 发撤销时，会先建孤儿记录再继续走撤销流程。

### 涉及代码
- `KeyManageController.cpp`：TasksUpdated 中的孤儿识别（line 1100-1107）
- `TicketStore.cpp`：`ingestOrphanTicket()`
- `KeyManageController.cpp`：cancel ingress 孤儿恢复路径（line 366-383）
