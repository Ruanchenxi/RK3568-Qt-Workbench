# 钥匙业务逻辑缺陷 — 修复执行方案

> 日期：2026-04-09
> 对应问题清单：[04_合并问题清单_2026-04-09.md](./04_合并问题清单_2026-04-09.md)
> 核心原则：保证现有正常自动链路（工作台传票→自动回传）的稳定性，修复不得引入新的副作用

---

## 分批策略

```
第一批（立即，改动极小）：C1、C5
第二批（核心，改动适中）：C3
第三批（优化，可延后）：  C8、C9、C6、C7
搁置（缺前置条件）：      C2、C4
```

### 搁置说明

**C2（撤销链钥匙身份校验）和 C4（传票反向对账）暂不执行。**

原方案用 `m_keySessionId`（串口会话编号）做"原钥匙"标识，但 session 不是钥匙身份——同一把钥匙拔了再插 session 就变了。用 session 做校验会误拒合法操作。

要正确解决 C2，需要协议层提供钥匙物理身份（如钥匙 UID/序列号）。在获得可靠的钥匙标识之前，C2 和依赖它的 C4 都不具备安全实施条件。

#### 设计说明

`transferSessionId` 记录的是"传票成功时的 keySessionId"。它的含义是：**这张票最后一次传到钥匙时，底座上是哪个 session**。

当钥匙拿走再放回（不管是同一把还是不同的），session 会变。这意味着：

- 钥匙没动过 → session 一致 → 撤销正常执行
- 钥匙拔走过 → session 不一致 → **撤销被拒绝**，提示"钥匙已更换，无法确认任务状态"

这是**安全保守的行为**：系统无法区分"同一把钥匙拔了又放回"和"换了不同的钥匙"，所以一律拒绝自动撤销。如果管理员确认钥匙状态后需要处理，可以用手工删除链路（C1/C3/C8）。

#### 具体改动

在 `reconcileSystemTicketsFromKeyTasks()` 函数末尾，原有正向对账循环之后，增加：

```cpp
// 反向对账：本地 success 但钥匙里不存在 → 回滚为 failed
const QList<SystemTicketDto> allTickets = m_ticketStore->tickets();
for (const SystemTicketDto &ticket : allTickets) {
    if (ticket.transferState != QLatin1String("success")) {
        continue;
    }
    // 已进入回传/撤销/删除流程的不回滚
    if (ticket.returnState != QLatin1String("idle") && !ticket.returnState.isEmpty()) {
        continue;
    }
    if (!ticket.cancelState.isEmpty() && ticket.cancelState != QLatin1String("none")) {
        continue;
    }
    // 不是当前 session 传票的不回滚（防止拿新钥匙数据误判旧票）
    if (ticket.transferSessionId > 0 && ticket.transferSessionId != m_keySessionId) {
        continue;
    }
    bool foundInKey = false;
    for (const KeyTaskDto &task : tasks) {
        if (taskIdFromRaw(task.taskId) == ticket.taskId) {
            foundInKey = true;
            break;
        }
    }
    if (!foundInKey) {
        m_ticketStore->updateTransferState(ticket.taskId,
                                            QStringLiteral("failed"),
                                            QStringLiteral("传票后验证：钥匙中未找到该任务，可能写入失败，请重新传票"));
        emit statusMessage(QStringLiteral("传票后验证失败：钥匙中未找到任务 %1").arg(ticket.taskId));
    }
}
```

#### 前提依赖

#### 为什么不影响其他功能

- 只在 success 且钥匙里确实没有 且 是当前 session 的情况下回滚
- 已进入回传/撤销的票全部跳过
- 回滚为 `failed`（不是删除），管理员可以看到并重传
- 换了钥匙的票被 `transferSessionId` 检查跳过，不会误判

## P0 修复：transTicket 延迟响应（同步确认票在钥匙后才返回 success）

### 问题背景

当前 `transTicket` 接口采用"先入池、立即返回 success=true"的设计（P0）。工作台收到 success 后隐藏传票按钮、显示"执行中"状态，用户可能直接拿走钥匙。但此时票可能尚未真正写入钥匙（串口传票异步进行），导致用户拿着没有票的钥匙去作业——在安全管控系统中这是不可接受的。

### 目标行为

```
工作台调 transTicket
→ 设备入本地票池（received），socket 挂起不回包
→ 串口传票完成（TicketTransferFinished）
→ 自动触发 Q_TASK 查询钥匙
→ Q_TASK 确认钥匙里有这张票
→ 回包 success=true，msg="传票成功，已确认任务存在于钥匙中"

任意环节失败 / 超时（8秒）：
→ 回包 success=false，msg=具体原因
→ 本地票标 failed，工作台传票按钮恢复
```

### 前提确认

- 工作台与设备本地局域网直连，无网络抖动风险
- 当前串口传票正常约 1 秒完成，Q_TASK 约 1~2 秒，合计 2~3 秒以内
- 超时设定 **8 秒**（约 2~3 倍正常耗时，足够安全）
- 工作台 HTTP 超时配置需大于 8 秒（本地连接默认值通常 30 秒以上，无需修改）

#### ③ TicketIngressService.cpp — `completeDeferredResponse()` 实现

```cpp
void TicketIngressService::completeDeferredResponse(const QString &taskId,
                                                     bool success,
                                                     const QString &message)
{
    auto it = m_deferredRequests.find(taskId);
    if (it == m_deferredRequests.end()) {
        return; // 已超时或已完成
    }
    QTcpSocket *socket = it->socket;
    QTimer     *timer  = it->timer;
    m_deferredRequests.erase(it);
    timer->stop();
    timer->deleteLater();

    const int sc = success ? 200 : 200; // HTTP 层始终 200，业务结果靠 success 字段
    emit httpServerLogAppended(
        QStringLiteral("[HTTP-INGRESS] deferred response taskId=%1 success=%2 msg=%3")
            .arg(taskId, success ? QStringLiteral("true") : QStringLiteral("false"), message));
    sendResponse(socket, sc, "OK",
                 buildJsonResponseBody(success, message, taskId, sc));
}
```

#### ④ TicketIngressService.cpp — `expireDeferredRequest()` 实现

```cpp
void TicketIngressService::expireDeferredRequest(const QString &taskId)
{
    auto it = m_deferredRequests.find(taskId);
    if (it == m_deferredRequests.end()) {
        return;
    }
    QTcpSocket *socket = it->socket;
    QTimer     *timer  = it->timer;
    m_deferredRequests.erase(it);
    timer->deleteLater();

    emit httpServerLogAppended(
        QStringLiteral("[HTTP-INGRESS] deferred transTicket timeout taskId=%1").arg(taskId));
    sendResponse(socket, 200, "OK",
                 buildJsonResponseBody(false,
                                       QStringLiteral("传票超时（8秒），请确认钥匙已就绪后重试"),
                                       taskId, 200));
}
```

#### ⑦ KeyManageController.cpp — `reconcileSystemTicketsFromKeyTasks()`

在正向对账循环中，当确认某个任务已在钥匙里（`transferState` 从非 success 纠正为 success）时，或已是 success 状态且本次 Q_TASK 确认存在时，加：

```cpp
// 如果有挂起的 HTTP 传票响应在等待这个 taskId，现在可以回包
m_ticketIngress->completeDeferredResponse(
    taskId, true,
    QStringLiteral("传票成功，已确认任务存在于钥匙中"));
```

**同时**，在 Q_TASK 回来后，如果某个任务有挂起的 deferred response 但 Q_TASK 里找不到这张票（即传票失败后 Q_TASK 扫描不到），补一个失败回包。可在 `reconcileSystemTicketsFromKeyTasks()` 末尾遍历 `m_ticketIngress` 的挂起列表：

```cpp
// Q_TASK 完成后，清理所有仍在等待但钥匙里找不到的挂起传票响应
// （说明 TicketTransferFinished 未触发，但 Q_TASK 确认无此任务）
// 由 TicketIngressService 的超时机制兜底，此处可不主动处理
// 若要提前失败，可在此遍历 m_ticketIngress->deferredTaskIds() 做检查
```

> **注意**：Q_TASK 触发后 `TicketTransferFailed` 通常已经处理了串口失败的情况。`reconcileSystemTicketsFromKeyTasks` 只需处理"传票成功但 Q_TASK 是第一个确认点"的场景。超时（8秒）作为最终兜底。

---

### 关键约束

1. **`auto-pending` 重试不能回 false**：自动传票失败转 auto-pending 后会自动重试，deferred socket 继续等待，由最终的 `TicketTransferFinished` + Q_TASK 完成
2. **`completeDeferredResponse` 幂等**：同一 taskId 第一次调用生效，后续调用直接忽略（已从 map 中移除）
3. **超时后 KeyManageController 的 completeDeferredResponse 调用无害**：此时 map 里已无记录，函数直接 return
4. **不影响非 HTTP 传票路径**：手工传票（`onTransferSelectedTicket`）不经过 ingress handler，不设置 deferred，不受影响
