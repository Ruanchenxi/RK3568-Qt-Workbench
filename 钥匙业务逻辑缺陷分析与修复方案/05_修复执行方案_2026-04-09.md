# 钥匙业务逻辑缺陷 — 修复执行方案

> 日期：2026-04-09
> 对应问题清单：[04_合并问题清单_2026-04-09.md](./04_合并问题清单_2026-04-09.md)
> 核心原则：保证现有正常自动链路（工作台传票→自动回传）的稳定性，修复不得引入新的副作用

---

## 分批策略

```
第一批（立即，改动极小）：C1、C5
第二批（核心，改动适中）：C3
第三批（优化，可延后）：  C8、C9、C6、C7
搁置（缺前置条件）：      C2、C4
```

### 搁置说明

**C2（撤销链钥匙身份校验）和 C4（传票反向对账）暂不执行。**

原方案用 `m_keySessionId`（串口会话编号）做"原钥匙"标识，但 session 不是钥匙身份——同一把钥匙拔了再插 session 就变了。用 session 做校验会误拒合法操作。

要正确解决 C2，需要协议层提供钥匙物理身份（如钥匙 UID/序列号）。在获得可靠的钥匙标识之前，C2 和依赖它的 C4 都不具备安全实施条件。

#### 设计说明

`transferSessionId` 记录的是"传票成功时的 keySessionId"。它的含义是：**这张票最后一次传到钥匙时，底座上是哪个 session**。

当钥匙拿走再放回（不管是同一把还是不同的），session 会变。这意味着：

- 钥匙没动过 → session 一致 → 撤销正常执行
- 钥匙拔走过 → session 不一致 → **撤销被拒绝**，提示"钥匙已更换，无法确认任务状态"

这是**安全保守的行为**：系统无法区分"同一把钥匙拔了又放回"和"换了不同的钥匙"，所以一律拒绝自动撤销。如果管理员确认钥匙状态后需要处理，可以用手工删除链路（C1/C3/C8）。

#### 具体改动

在 `reconcileSystemTicketsFromKeyTasks()` 函数末尾，原有正向对账循环之后，增加：

```cpp
// 反向对账：本地 success 但钥匙里不存在 → 回滚为 failed
const QList<SystemTicketDto> allTickets = m_ticketStore->tickets();
for (const SystemTicketDto &ticket : allTickets) {
    if (ticket.transferState != QLatin1String("success")) {
        continue;
    }
    // 已进入回传/撤销/删除流程的不回滚
    if (ticket.returnState != QLatin1String("idle") && !ticket.returnState.isEmpty()) {
        continue;
    }
    if (!ticket.cancelState.isEmpty() && ticket.cancelState != QLatin1String("none")) {
        continue;
    }
    // 不是当前 session 传票的不回滚（防止拿新钥匙数据误判旧票）
    if (ticket.transferSessionId > 0 && ticket.transferSessionId != m_keySessionId) {
        continue;
    }
    bool foundInKey = false;
    for (const KeyTaskDto &task : tasks) {
        if (taskIdFromRaw(task.taskId) == ticket.taskId) {
            foundInKey = true;
            break;
        }
    }
    if (!foundInKey) {
        m_ticketStore->updateTransferState(ticket.taskId,
                                            QStringLiteral("failed"),
                                            QStringLiteral("传票后验证：钥匙中未找到该任务，可能写入失败，请重新传票"));
        emit statusMessage(QStringLiteral("传票后验证失败：钥匙中未找到任务 %1").arg(ticket.taskId));
    }
}
```

#### 前提依赖

#### 为什么不影响其他功能

- 只在 success 且钥匙里确实没有 且 是当前 session 的情况下回滚
- 已进入回传/撤销的票全部跳过
- 回滚为 `failed`（不是删除），管理员可以看到并重传
- 换了钥匙的票被 `transferSessionId` 检查跳过，不会误判

## P0 修复：transTicket 延迟响应（同步确认票在钥匙后才返回 success）

### 问题背景

当前 `transTicket` 接口采用"先入池、立即返回 success=true"的设计（P0）。工作台收到 success 后隐藏传票按钮、显示"执行中"状态，用户可能直接拿走钥匙。但此时票可能尚未真正写入钥匙（串口传票异步进行），导致用户拿着没有票的钥匙去作业——在安全管控系统中这是不可接受的。

### 目标行为

```
工作台调 transTicket
→ 设备入本地票池（received），socket 挂起不回包
→ 串口传票完成（TicketTransferFinished）
→ 自动触发 Q_TASK 查询钥匙
→ Q_TASK 确认钥匙里有这张票
→ 回包 success=true，msg="传票成功，已确认任务存在于钥匙中"

任意环节失败 / 超时（8秒）：
→ 回包 success=false，msg=具体原因
→ 本地票标 failed，工作台传票按钮恢复
```

### 前提确认

- 工作台与设备本地局域网直连，无网络抖动风险
- 当前串口传票正常约 1 秒完成，Q_TASK 约 1~2 秒，合计 2~3 秒以内
- 超时设定 **8 秒**（约 2~3 倍正常耗时，足够安全）
- 工作台 HTTP 超时配置需大于 8 秒（本地连接默认值通常 30 秒以上，无需修改）

## 第五批：P1 — 删除后通知工作台时序修复（2026-04-10 审查修订版）

> 业务描述：管理员点"删除钥匙票"，DEL 成功后设备需通知工作台"任务已终止"。当前实现是通知发出后**立即**删除本地系统票记录，不等工作台确认。一旦通知失败，本地记录已删，设备无法补发通知，工作台任务永远挂着。

### 审查修订要点（相对旧版）

1. **必须持久化，不能只靠内存**：旧方案用 `QSet<QString>` 在内存里跟踪在途 termination，掉电/崩溃会丢失"钥匙已清、工作台未终止"这个关键事实。新方案把中间态落到 `ticket_store.json`。
2. **不借用 `returnState`**：旧方案把 HTTP 失败塞进 `returnState = "return-delete-failed"`。手工删除链从来没进过回传流程，污染 returnState 会让后续 reconcile / auto-return / canStartTicketReturn 全部误判。新方案使用独立字段。
3. **新增 `adminDeleteStage` 枚举字段**：单独表达管理员删除链的三个状态，回传链完全不感知此字段。

### 持久化字段设计

## 第六批：P2 — 传票确认用时间戳字段（低风险版，2026-04-10 审查修订）

> 业务描述：串口协议 `TicketTransferFinished` 报告"传票数据发完了"后，系统立即把票标记为 `transferState = "success"`。但串口 ACK 不等于钥匙真的写入成功——个别场景下钥匙未真正接收，本地却已显示成功。此状态会误导后续的自动回传 / 手动回传 / 撤销链，可能导致链路卡死。

### 审查修订要点（相对旧版）

**放弃旧版"改 transferState 主语义"的方案**。旧版要引入 `transfer-verifying` 新状态值，会波及 **6 处** `transferState == "success"` 判断点，改一半会引入新卡死；且降级机制依赖 P0 的 deferred expire 信号，无法覆盖手工传票和自动传票（两者都没有 deferred socket）。

**改为：主语义保留，另加时间戳字段**。`transferState = "success"` 继续表示"串口说传完了"；另加 `keyConfirmedAtMs` 表示"最近一次 Q_TASK 亲眼确认此票在钥匙里的时间戳"。只有真正依赖"钥匙中确实有"的业务判断点才检查新字段（**只改 3 处**）。降级不依赖 HTTP，与传票来源无关。

### 设计思路

```
场景 A：串口传票完成 + Q_TASK 亲眼确认
  → transferState = "success"
  → keyConfirmedAtMs = 最近一次 Q_TASK 的时间戳
  → isKeyPresenceFresh() = true
  → 自动回传 / 手动回传正常进行

场景 B：串口说成功 但钥匙实际没写进去
  → transferState = "success"（串口说的）
  → keyConfirmedAtMs 落后于 m_lastQueryTasksAtMs（Q_TASK 没看到）
  → isKeyPresenceFresh() = false
  → 自动回传 / 手动回传跳过这张票，不卡链
  → 由单条清理或管理员手工处理

场景 C：钥匙被取走
  → Q_TASK 跑时 lastKeyTasks 为空 → reconcile 不刷新票的时间戳
  → keyConfirmedAtMs 陈旧
  → isKeyPresenceFresh() = false → 保护生效
```

`KeyManageController` 新增成员与 helper：

```cpp
qint64 m_lastQueryTasksAtMs = 0;  // 最近一次成功 Q_TASK 的时间戳（无论钥匙里有几个任务）

bool isKeyPresenceFresh(const SystemTicketDto &ticket) const
{
    if (ticket.transferState != QLatin1String("success")) return false;
    if (ticket.keyConfirmedAtMs <= 0)                     return false;
    if (m_lastQueryTasksAtMs <= 0)                        return false;
    // 关键：此票的确认时间戳不早于最近一次 Q_TASK（给 500ms 缓冲），就算新鲜
    return ticket.keyConfirmedAtMs + 500 >= m_lastQueryTasksAtMs;
}
```

**判定逻辑含义**：每跑一次 Q_TASK，`m_lastQueryTasksAtMs` 刷新到当前时间；Q_TASK 里看到的票的 `keyConfirmedAtMs` 也刷新到当前时间。如果某张 success 票上一次 Q_TASK 没被看到，它的时间戳会停留在之前的值，落后于 `m_lastQueryTasksAtMs`，从而判定为"不新鲜"——保护生效。无需固定 TTL 数字。

### 子步骤 3：单条清除钥匙中未登记任务

> 业务描述：钥匙里存有某条任务，但本机 `ticket_store` 中没有对应记录。这类"钥匙未登记任务"通常由历史遗留 / 异常重启造成。设备本地无法证明工作台有没有这条记录，因此此操作**只做设备侧清理**，不调 termination。

#### 触发方式

在"钥匙任务列表"的每一行：
- 如果本行对应的 taskId **不存在于 ticket_store** → 视为"未登记"，行内显示"清除此项"按钮
- 按钮仅对管理员可见（非管理员不显示）

#### 前置条件

- 串口已连接且钥匙在位
- 当前无传票 / 删除 / 回传 / 撤销在途（通过 `m_activeTransferTaskId` / `m_pendingDeletedSystemTicketId` / `m_activeReturnTaskId` / `m_pendingCancelSystemTicketId` 判断）
- 任一条件不满足 → 按钮 disable 并 tooltip 说明原因

#### 执行流程

```
点击 → 确认对话框（显示 taskId + "此任务在本机无登记，是否从钥匙中清除？"）
→ 管理员确认 → 单条 DEL 串口命令
→ 收到 KeyTaskDeleted 事件 → 成功提示，刷新钥匙任务列表
→ 超时 / 失败 → 错误提示
```

**不调 termination**——设备不知道工作台是否有此任务。

#### 改动范围

| 文件 | 改动 |
|------|------|
| `keymanagepage.cpp` | 钥匙任务列表渲染中：判定每行是否"未登记"，对未登记行追加"清除此项"按钮 |
| `keymanagepage.cpp` | 新增 `onClearResidualKeyTaskRow(taskId)` slot |
| `KeyManageController.h/.cpp` | 新增 `bool clearResidualKeyTask(const QString &taskId, QString *errorOut)`：内部走前置检查 + 串口 DEL + 单条等待回包 |

### 子步骤 4：单条强制清理失败系统票

> 业务描述：`transferState == "failed"` 的票无法自动恢复，管理员希望从系统票池中强制清除并通知工作台终止。第一版**只处理 failed 一种状态**。

#### 触发方式

在"系统票池列表"的每一行，满足以下全部条件时显示"强制清理"按钮：
- `transferState == "failed"`
- `cancelState` 为空或 `"none"`
- `returnState` 未进入任何清理流程（空 / `"idle"`）
- `adminDeleteStage == None`（不与 P1 在途流程冲突）
- 仅对管理员可见

#### 前置条件

- 当前无本票相关的删除 / 回传 / 撤销在途
- 可与其他 taskId 的在途操作并行（单条操作不抢主状态机）

#### 执行流程

```
点击 → 确认对话框（"此票已传票失败，强制清理将通知工作台终止并删除本机记录。继续？"）
→ 管理员确认 → 复用 P1 的 callTerminationThenRemove(taskId)
→ HTTP 成功：本机记录删除 + 提示
→ HTTP 失败：本机保留 + 错误提示（与 P1 同样路径）
```

**关键依赖**：此步骤依赖 P1 已完成（`callTerminationThenRemove` 和 `adminDeleteStage` 字段必须先就绪）。

| `KeyManageController.h/.cpp` | 新增 `bool forceCleanFailedSystemTicket(const QString &taskId, QString *errorOut)`：前置检查通过后调 `callTerminationThenRemove` |
