## P3 修复方案：DEL失败后自动重试

### 为什么先修P3

P3是板端实测直接复现的问题，修复方案最明确，可以先用它验证修复流程。

### 改动范围

| 文件 | 改动 |
|------|------|
| `KeyManageController.cpp` | `finalizePendingReturnDelete()` 的 `stillExists` 分支 |

### 具体方案

在 `finalizePendingReturnDelete()` 中，当 `stillExists == true` 时，不再只打日志，而是**重新发送 DEL 命令**：

```
原逻辑：
  stillExists == true → 打日志"等待后续自动清理" → 什么都不做

修改为：
  stillExists == true
    → 检查钥匙是否在位且通讯正常
    → 是 → 重新构造 DEL CommandRequest，调用 m_session->execute()
    → 否 → 打日志"钥匙不在位，等待放回后重试"（保留现有行为）
```

### 防止无限重试

增加重试计数器 `m_pendingDeleteRetryCount`：
- 每次重发 DEL，计数器 +1
- 上限设为 3 次
- 超过上限 → 状态转为 `return-delete-failed`，UI 显示"删除失败，请手动处理"
- 钥匙被拔走再放回时，重置计数器（新的物理周期）

### 验证方法

1. 传票 → 回传 → DEL发出后迅速拔钥匙
2. 放回钥匙，观察是否自动重发DEL
3. 确认Q_TASK显示 count=0 后状态变为 `return-delete-success`

## P1 修复方案：撤销请求增加物理状态检查

### 改动范围

| 文件 | 改动 |
|------|------|
| `KeyManageController.cpp` | cancel ingress handler（约 322~464 行） |

### 具体方案

在 cancel ingress handler 的**兜底分支之前**（约 428 行），增加物理状态守卫：

```
新增判断：
  如果 ticket.transferState == "success"
    且 钥匙不在位（!m_keyPresent 或 m_lastKeyTasks 为空）
    → 拒绝撤销
    → result.accepted = false
    → result.message = "任务已下发到钥匙且钥匙已被取走，无法撤销正在执行的任务"
    → 返回 HTTP 4xx / 业务拒绝码给工作台
```

### 补充：钥匙在位但任务已传入的撤销

如果钥匙在位（还没拿走），撤销是合理的——操作还没开始。这种情况保持现有逻辑：受理撤销 → 等 Q_TASK 对账 → DEL 删除。

### 判断条件总结

```
钥匙不在位 + transferState=success     → 拒绝撤销（操作员可能在现场）
钥匙在位   + transferState=success     → 受理撤销（操作还没开始）
钥匙不在位 + transferState=received    → 受理撤销（还没传到钥匙，直接移除）
任何状态   + returnState 已在回传链路中 → 保持现有逻辑（已在清理中）
```

### 验证方法

1. 传票到钥匙 → 拔走钥匙 → 工作台发撤销 → 确认被拒绝
2. 传票到钥匙 → 钥匙在位 → 工作台发撤销 → 确认被受理并执行DEL
3. 任务还没传到钥匙 → 撤销 → 确认直接移除

## P2 修复方案：换钥匙后不用新钥匙的数据对账旧任务

### 改动范围

| 文件 | 改动 |
|------|------|
| `KeyManageController.cpp` | `finalizePendingReturnDelete()`（约 1688~1735 行） |
| `KeyManageController.cpp` | link-interrupted handler（约 1357~1365 行） |

### 具体方案

修改 session 检查逻辑，**不再用 0 跳过检查**：

```
原逻辑：
  if (m_pendingDeleteSessionId > 0 && !isCurrentKeySession(...)) → skip

修改为：
  // sessionId == 0 说明钥匙被拔走过，必须确认是同一把钥匙才能继续对账
  if (m_pendingDeleteSessionId == 0) {
      // 钥匙曾被移除，无法确认当前钥匙是否是同一把
      // 不做 false-positive 判定，改为重新发 DEL
      → 走 P3 的重试逻辑
  }
  if (m_pendingDeleteSessionId > 0 && !isCurrentKeySession(...)) {
      → skip（不同 session，等同一把钥匙回来）
  }
```

### 核心原则

**宁可多删一次（DEL 是幂等的），不可误判删除成功。**

如果无法确认当前钥匙就是发过 DEL 的那把钥匙，就重新发一次 DEL。DEL 对不存在的 taskId 发送后钥匙会返回 ACK（幂等），不会有副作用。

### 验证方法

1. 钥匙A传票 → 回传 → DEL失败 → 拔走钥匙A → 插入钥匙B
2. 确认不会对钥匙B误判删除成功
3. 重新插入钥匙A → 确认自动重发DEL并最终成功

## P4 修复方案：传票成功时机延迟到Q_TASK验证后

### 改动范围

| 文件 | 改动 |
|------|------|
| `KeyManageController.cpp` | `TicketTransferFinished` handler（约 1197~1213 行） |
| `KeyManageController.cpp` | `TasksUpdated` handler / `reconcileSystemTicketsFromKeyTasks()`（约 1974~1999 行） |

### 具体方案

```
原逻辑：
  TICKET帧全部ACK → 标记 success → 发 Q_TASK 验证（验证结果不影响状态）

修改为：
  TICKET帧全部ACK → 标记 transfer-verifying（新增中间状态）→ 发 Q_TASK
    → Q_TASK 确认任务在钥匙中 → 标记 success
    → Q_TASK 未发现任务         → 标记 transfer-failed，提示重新传票
    → Q_TASK 超时               → 标记 success（降级为信任ACK）
```

### 新增状态

在 transferState 中新增 `transfer-verifying`：

- 含义：TICKET帧已全部ACK，正在等待Q_TASK确认
- 超时策略：5秒内Q_TASK未返回 → 降级为 success（ACK本身已是较强保证）
- reconcile 需要识别此状态

### 验证方法

1. 正常传票 → 确认先进入 `transfer-verifying` → Q_TASK确认后变为 `success`
2. 模拟钥匙存储异常（如果可能）→ 确认 Q_TASK 发现任务不在时标记 `transfer-failed`

### 注意事项

- 此修改影响面较大，涉及状态机新增状态，需要同步更新 TicketStore 的状态判断
- 建议在 P3/P1/P2 修复并验证后再做此项
- 如果评估风险过高，可降级为：保持现有 success 时机，但在 reconcile 中增加对 success 状态的检查（如果 Q_TASK 不含该任务则标记异常）

## P5 修复方案：HTTP回传增加超时保护

### 改动范围

| 文件 | 改动 |
|------|------|
| `TicketReturnHttpClient.cpp`（或对应HTTP请求类） | 增加请求超时 |
| `KeyManageController.cpp` | 增加应用层超时兜底 |

### 具体方案

**双层超时**：

#### 第一层：HTTP请求本身的超时

```cpp
// QNetworkRequest 设置超时（Qt 5.15+）
request.setTransferTimeout(30000); // 30秒
```

或者如果 Qt 版本不支持 `setTransferTimeout`：

```cpp
// 用 QTimer 手动超时
QTimer::singleShot(30000, this, [this, taskId]() {
    if (m_activeReturnTaskId == taskId) {
        handleReturnUploadFailed(taskId, "HTTP请求超时");
    }
});
```

#### 第二层：KeyManageController 应用层兜底

在 `tryStartTicketReturn()` 中启动一个兜底定时器：

```
发起HTTP上传时启动 m_returnUploadTimeoutTimer（60秒）
→ 超时触发：
  → 调用 handleReturnUploadFailed(taskId, "回传超时")
  → 状态转为 return-upload-timeout（可重试状态）
  → 清除 m_activeReturnTaskId，解除阻塞
```

正常回调到来时取消定时器。

### 验证方法

1. 断开网络 → 执行回传 → 确认30秒后自动超时、状态正确、不阻塞后续操作
2. 恢复网络 → 再次回传 → 确认正常完成

### 具体方案

```cpp
// 如果该任务正在撤销链路中，不抢占
if (!m_pendingCancelSystemTicketId.isEmpty()
    && m_pendingCancelSystemTicketId == taskId) {
```

## P7 修复方案：延迟命令槽改为队列

### 改动范围

| 文件 | 改动 |
|------|------|
| `KeySerialClient.cpp` | `ensureBusinessReadyOrDefer()`（约 2124~2148 行） |
| `KeySerialClient.h` | 将单槽位改为 `QQueue<DeferredCommand>` |

### 具体方案

```
原逻辑：
  m_deferredInFlightType = type;  // 后来的覆盖前面的

修改为：
  struct DeferredCommand {
      int type;
      QByteArray payload;
  };
  QQueue<DeferredCommand> m_deferredCommands;  // FIFO队列

  // defer 时入队
  m_deferredCommands.enqueue({type, payload});

  // 钥匙就绪时，逐个出队执行
  while (!m_deferredCommands.isEmpty()) {
      auto cmd = m_deferredCommands.dequeue();
      // 执行...（每次只执行一个，等ACK后再出下一个）
      break;
  }
```

### 队列上限

设置最大 4 个，超出时丢弃最旧的并打 WARN 日志。

### 验证方法

1. 钥匙稳定中连续触发 Q_TASK + DEL → 确认两个命令都被执行

## P8 修复方案：撤销日志增加过期清理

### 改动范围

| 文件 | 改动 |
|------|------|
| `TicketCancelCoordinator.cpp` | `loadRecords()` 或新增 `purgeExpired()` |

### 具体方案

```cpp
void TicketCancelCoordinator::purgeExpired()
{
    const QDateTime cutoff = QDateTime::currentDateTime().addDays(-7);
    auto it = m_records.begin();
    while (it != m_records.end()) {
        if (isTerminalState(it->cancelState) && it->timestamp < cutoff) {
            it = m_records.erase(it);
        } else {
            ++it;
        }
    }
    saveRecords();
}
```

在 `loadRecords()` 末尾调用 `purgeExpired()`，每次启动时自动清理 7 天前的终态记录。

### 验证方法

1. 手动在 journal.json 中写入过期记录 → 重启 → 确认被清理

## P9 修复方案：孤儿恢复降级为“待人工确认”，禁止直接自动接管并自动回传

### 改动范围

| 文件 | 改动 |
|------|------|
| `KeyManageController.cpp` | `TasksUpdated` 中 orphan recovery 分支（约 1073~1084 行） |
| `TicketStore.cpp` | `ingestOrphanTicket()`（约 199~230 行） |
| `KeyManageController.cpp` | `tryAutoReturnCompletedTicket()`（约 2154~2177 行） |

### 具体方案

当前孤儿恢复过于激进，应该改成“恢复为待人工确认对象”，而不是直接恢复成正式成功票。

```
原逻辑：
  钥匙里有任务 + 本地没有
    → ingestOrphanTicket(taskId)
    → transferState=success, returnState=idle
    → 后续可能自动回传

修改为：
  钥匙里有任务 + 本地没有
    → ingestOrphanTicket(taskId)
    → transferState=orphan-recovered
    → returnState=manual-required
    → 不参与自动回传
    → UI 明确提示“发现未知钥匙任务，请人工确认来源后处理”
```

### 核心原则

- 可以保留“孤儿恢复”能力，避免程序重启后彻底丢任务
- 但不能直接把未知任务当成“本机合法成功票”
- 尤其不能在未确认来源前参与自动回传或自动删除

### 建议落地细节

1. `ingestOrphanTicket()` 新建时不要直接写 `success/idle`
2. `tryAutoReturnCompletedTicket()` 跳过：
   - `transferState == orphan-recovered`
   - 或 `returnState == manual-required`
3. UI 层把这类票显式标成：
   - “孤儿恢复任务”
   - “需人工确认”

### 验证方法

1. 插入一把带有未知任务的钥匙
2. 确认本地只出现“待人工确认”的恢复记录
3. 确认不会自动回传
4. 确认不会自动清理钥匙中的该未知任务

## P10 修复方案：明确区分“人工失败”和“自动待重试”，禁止人工失败静默转 auto-pending

### 改动范围

| 文件 | 改动 |
|------|------|
| `KeyManageController.cpp` | `TicketTransferFailed` 分支（约 1236~1250 行） |
| `KeyManageController.cpp` | `recoverAutoTransferWhenKeyEmpty()`（约 2002~2046 行） |
| `SystemTicketDto` / `TicketStore` | 如有必要，补充触发来源标记 |

### 具体方案

```
原逻辑：
  人工传票失败
    → 某些分支会改成 auto-pending
    → 后续自动重发

修改为：
  如果本次传票是人工触发
    → 失败后统一保持 failed
    → 不转 auto-pending
    → 只能由用户再次点击重试

  如果本次传票是自动触发
    → 才允许转 auto-pending 并自动重试
```

### 建议实现方式

给当前活动传票上下文增加“触发来源”：

- `manual`
- `automatic`

然后：

1. `TicketTransferFailed` 的 inflightBusy 分支先判断来源
2. `recoverAutoTransferWhenKeyEmpty()` 在回队前也判断来源
3. 只有自动来源的票允许重新进入 `auto-pending`

### 核心原则

**人工失败 = 明确失败，必须人工重试。**

不要再出现：
- 这次人工点击失败了
- 系统过一会儿自己又发成功了

### 验证方法

1. 人工点击传票后制造失败
2. 确认状态停在 `failed`
3. 不再点击的情况下，确认系统不会自动补发
4. 再次人工点击，确认可以重新发送

## P11 修复方案：HTTP传票安全超时时彻底清理活动传票上下文

### 改动范围

| 文件 | 改动 |
|------|------|
| `KeyManageController.cpp` | HTTP 事务安全超时回调（约 123~135 行） |
| `KeyManageController.cpp` | `clearHttpTransferTransaction()`（约 2411~2435 行） |

### 具体方案

```
原逻辑：
  60 秒超时
    → clearHttpTransferTransaction(false)
    → 只清 HTTP 表层锁
    → activeTransfer / handshake 缓存可能残留

修改为：
  60 秒超时
    → clearHttpTransferTransaction(true)
    → 一并清理：
       - m_activeTransferTaskId
       - m_activeTransferSessionId
       - m_pendingTransferHandshakeTaskId
       - m_pendingTransferHandshakeJsonBytes
       - m_pendingTransferHandshakeDebugChunk
    → 同时把对应票状态改为 failed / timeout
```

### 核心原则

既然日志已经说“强制释放”，那业务上下文也必须真正释放干净。

不能出现：
- 日志显示已解锁
- 后续传票仍被 `m_activeTransferTaskId` 挡住

### 建议落地细节

1. 安全超时回调里，除了 `clearHttpTransferTransaction(true)` 外
2. 如果仍有对应 `taskId`
   - `updateTransferState(taskId, "failed", "HTTP传票事务超时，已强制清理")`
3. 补一条明确日志：
   - “活动传票上下文已一并清理”

### 验证方法

1. 人为制造一条卡住 60 秒的 HTTP 传票
2. 超时后确认：
   - 锁已释放
   - `m_activeTransferTaskId` 已清空
   - 后续新传票能正常再次发起

## 修复顺序与依赖关系

```
P3（DEL重试）──────┐
                   ├── P2 依赖 P3 的重试机制
P1（撤销守卫）──── │── 独立，可与 P3 并行
                   │
P2（session检查）──┘

P5（HTTP超时）──── 独立，可与上面并行

P4（传票验证延迟）── 独立但改动面大，建议放在第一批末尾

P9（孤儿恢复降级）── 独立，建议放在第一批前中段
P10（人工失败不自动补发）── 与自动传票逻辑强相关，建议第一批中段
P11（安全超时真释放）── 独立，建议第一批前中段

P6（清理/撤销互斥）── 独立，第二批
P7（延迟命令队列）── 独立，第二批
P8（日志过期）──── 独立，第三批
```

### 建议实施顺序

```
1. P3 → P2（DEL重试 + session检查，解决板端实测问题）
2. P1（撤销守卫，最严重的安全问题）
3. P9（孤儿恢复降级，防止接管外来任务）
4. P10（人工失败禁止静默补发）
5. P11（HTTP传票超时真释放）
6. P5（HTTP回传超时，防止系统锁死）
7. P4（传票验证延迟，需要谨慎评估）
8. P6 → P7 → P8（第二三批）
```
