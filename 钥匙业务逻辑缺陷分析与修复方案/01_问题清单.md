# 钥匙业务逻辑缺陷问题清单

> 审查日期：2026-04-07
> 审查范围：KeyManageController / TicketStore / TicketCancelCoordinator / KeySerialClient
> 触发背景：板端实测发现"回传完成后钥匙任务一直删不掉"，顺藤摸瓜发现一系列状态机和业务逻辑漏洞

## P1：撤销不检查钥匙物理状态（严重）

### 现象

工作台对一个"钥匙已被拿走、操作员正在现场执行停送电"的任务发起撤销 → 程序受理 → 等钥匙放回后删除任务。

### 根因

cancel ingress handler（`KeyManageController.cpp` 约 322~464 行）的判断逻辑：

1. 钥匙被拔走时 `invalidateCurrentKeySession()` 清空了 `m_lastKeyTasks`
2. `taskExistsInList(m_lastKeyTasks, taskId)` 返回 false
3. 如果 transferState 是 `received` / `auto-pending`，直接走 direct-remove 分支（414~425 行）——**直接从 TicketStore 中删除**
4. 如果 transferState 是 `success`，走兜底分支（428~464 行）——标记 `cancel-accepted`，等钥匙放回后执行

**两种情况都没有检查"钥匙是否在位"或"任务是否正在被执行"。**

### 业务影响

- 操作员在现场正操作 411 断路器，系统却把任务标记为"已撤销"
- 操作记录与物理操作脱节，停送电安全流程被破坏
- 如果后台因撤销而派另一个操作员去同一设备，可能两人同时操作——违反电力安全规程

### 涉及代码

- `KeyManageController.cpp`：cancel ingress lambda（约 322~464 行）
- `KeyManageController.cpp`：`invalidateCurrentKeySession()`（约 2875~2893 行）

## P2：换钥匙导致误判删除成功（严重）

### 现象

钥匙A回传后发DEL → 钥匙A被拔走（DEL失败）→ 插入钥匙B → Q_TASK查到钥匙B的任务列表（不含钥匙A的任务）→ 系统判定"删除成功"。

### 根因

`finalizePendingReturnDelete()`（`KeyManageController.cpp` 约 1688~1735 行）：

1. 钥匙被拔走时，`m_pendingDeleteSessionId` 被清为 0
2. session 检查条件是 `if (m_pendingDeleteSessionId > 0 && !isCurrentKeySession(...))`
3. 当 sessionId == 0 时，`> 0` 为 false，**整个检查被跳过**
4. 程序拿钥匙B的 Q_TASK 结果去找钥匙A的 taskIdRaw → 找不到 → 判定"已删除"

### 业务影响

- 系统标记 `return-delete-success`，工作台显示任务已完成
- 钥匙A上的任务实际还在，如果被其他人使用，可能执行未经授权的设备操作

### 涉及代码

- `KeyManageController.cpp`：`finalizePendingReturnDelete()`（约 1688~1735 行）
- `KeyManageController.cpp`：link-interrupted handler 中 `m_pendingDeleteSessionId = 0`（约 1357~1365 行）

## P3：DEL失败后不重试，永远卡在"正在对账"（高）

### 现象

回传HTTP上传成功 → 发送DEL → 钥匙在DEL发出167ms后被拔走 → DEL未收到ACK → 之后反复插拔钥匙，Q_TASK不断发现任务还在，但永远不重发DEL。

### 根因

`finalizePendingReturnDelete()` 的 `stillExists == true` 分支（约 1729~1733 行）：

```cpp
// 当前逻辑：只打日志，不重试
m_ticketStore->updateReturnState(..., "return-delete-verifying",
    "删除后验证发现钥匙任务仍存在，等待后续自动清理");
```

**"后续自动清理"在代码中不存在。** 没有任何路径会重新发送DEL命令。

### 实际日志证据

```
[TicketStore] return-delete-verifying -> return-delete-verifying error=删除后验证发现钥匙任务仍存在
[TicketStore] return-delete-verifying -> return-delete-verifying error=钥匙已移除，删除验证已中断
[TicketStore] return-delete-verifying -> return-delete-verifying error=删除后验证发现钥匙任务仍存在
... （无限循环）
```

### 业务影响

- 任务回传流程永远无法完成
- 工作台一直显示"正在对账确认钥匙任务是否已删除"
- 新任务无法传入该钥匙（pendingDelete 阻塞 tryAutoTransfer）

### 涉及代码

- `KeyManageController.cpp`：`finalizePendingReturnDelete()` 的 else 分支（约 1729~1735 行）
- `KeyManageController.cpp`：`handleReturnUploadSucceeded()`（约 1624~1644 行）
- `KeySerialClient.cpp`：`deleteTask()`（约 607~645 行）

## P4：传票标记success太早，验证失败不回滚（高）

### 现象

所有TICKET帧发送完毕、ACK收到 → 立刻标记 `transfer-success` → 然后才发Q_TASK验证 → 如果验证发现钥匙里没有这个任务，不会回滚 success 状态。

### 根因

`KeyManageController.cpp` 约 1197~1213 行：

1. `TicketTransferFinished` 事件触发 → 直接 `updateTransferState(taskId, "success")`
2. 然后触发 Q_TASK 验证
3. `reconcileSystemTicketsFromKeyTasks()` 只处理 `received/auto-pending/sending/failed` 状态的票，**不处理已经是 success 的票**

### 业务影响

- 如果钥匙存储满了或静默丢弃了数据，操作员拿着"空钥匙"去现场
- 系统认为传票成功，但钥匙实际不含任务数据
- 操作员无法在现场执行操作，需要返回重新传票

### 涉及代码

- `KeyManageController.cpp`：`TicketTransferFinished` handler（约 1197~1213 行）
- `KeyManageController.cpp`：`reconcileSystemTicketsFromKeyTasks()`（约 1974~1999 行）

## P5：HTTP回传无超时，单个请求锁死全部操作（高）

### 现象

回传日志HTTP上传发出 → 后端无响应或网络中断 → `m_activeReturnTaskId` 一直不清 → 所有后续回传和撤销被阻塞。

### 根因

`uploadReturnPayload()`（约 1150~1153 行）发出HTTP请求后，依赖 `handleReturnUploadSucceeded` 或 `handleReturnUploadFailed` 回调来清理状态。如果回调永远不来（网络挂起），状态永远不清。

钥匙不被拔走的情况下，没有任何超时机制来兜底。`canProcessCancelNow()` 检查 `activeReturnChainTaskId()` 非空就返回 false，阻塞一切。

### 业务影响

- 一个网络问题导致整个钥匙管理系统停摆
- 无法回传、无法撤销、无法传新票
- 必须重启程序才能恢复

### 涉及代码

- `KeyManageController.cpp`：`tryStartTicketReturn()` → `uploadReturnPayload()`（约 1150~1153 行）
- `KeyManageController.cpp`：`activeReturnChainTaskId()` / `canProcessCancelNow()`

## P6：自动清理DEL和撤销DEL竞争同一任务（中）

### 现象

一个任务同时处于 `return-upload-success`（回传链路）和 `cancel-accepted`（撤销链路），两条链路各自对它发DEL。

### 根因

`tryAutoCleanupReturnedTicket()`（约 2050~2093 行）检查了 `m_activeReturnTaskId` 和 `m_pendingDeletedSystemTicketId`，但**没有检查 `m_pendingCancelSystemTicketId`**，也没有检查票据的 cancelState。

### 业务影响

- 两条链路竞争，状态转换混乱
- 可能导致重复记账或状态不一致

### 涉及代码

- `KeyManageController.cpp`：`tryAutoCleanupReturnedTicket()`（约 2050~2093 行）

## P7：延迟命令槽只有一个，静默覆盖（中）

### 现象

钥匙稳定中，先触发Q_TASK延迟 → 紧接着触发DEL延迟 → Q_TASK被静默丢弃。

### 根因

`KeySerialClient.cpp` 的 `ensureBusinessReadyOrDefer()`（约 2124~2148 行）只有一个 `m_deferredInFlightType` + `m_deferredDeleteTaskId` 槽位，后来的覆盖前面的，无日志提示。

### 业务影响

- 跳过一次Q_TASK对账，可能导致状态不一致
- 操作被静默吞掉，难以排查

### 涉及代码

- `KeySerialClient.cpp`：`ensureBusinessReadyOrDefer()`（约 2124~2148 行）

## P8：撤销日志无过期清理（低）

### 现象

`cancel_ticket_journal.json` 只增不减。

### 根因

`TicketCancelCoordinator` 没有像 `TicketStore` 的 `isExpiredFinished()`（7天过期）一样的清理机制。终态记录永久保留。

### 业务影响

- 不影响业务正确性

### 涉及代码

- `TicketCancelCoordinator.cpp`：`loadRecords()` / `saveRecords()`（约 168~196 行）

## P9：孤儿恢复会直接接管未知钥匙任务，并可能自动回传/清理（严重）

### 现象

插入一把“本地票池里没有记录，但钥匙里仍有任务”的钥匙：

- 可能是程序重启后本地票池丢失
- 可能是另一台机器处理过的钥匙
- 可能是旧任务残留钥匙

程序执行 `Q_TASK` 后，会把这条任务直接补进本地票池，并按“已成功传到钥匙”的票处理；如果任务状态又是完成态，还可能继续触发自动回传和后续自动清理。

### 根因

`TasksUpdated` 分支（`KeyManageController.cpp` 约 1073~1084 行）在 ready 状态下会主动做孤儿恢复：

1. 遍历钥匙里的任务列表
2. 只要本地 `TicketStore` 里没有对应 `taskId`
3. 就直接调用 `ingestOrphanTicket(taskId)`

而 `TicketStore::ingestOrphanTicket()`（`TicketStore.cpp` 约 199~230 行）会把这条任务直接建成：

- `transferState = success`
- `returnState = idle`

后续 `tryAutoReturnCompletedTicket()`（`KeyManageController.cpp` 约 2154~2177 行）又会对：

- `transferState == success`
- `returnState` 未进入跳过态
- 钥匙中任务状态 `0x02`

的票直接触发自动回传。

### 业务影响

- 插入一把“不是本机本轮业务生成的钥匙”，系统也可能把它接管成自己的票
- 可能对外来任务执行自动回传、自动删除
- 容易把“残留任务”误处理成“本机合法完成任务”

### 涉及代码

- `KeyManageController.cpp`：`TasksUpdated` 中的 orphan recovery（约 1073~1084 行）
- `TicketStore.cpp`：`ingestOrphanTicket()`（约 199~230 行）
- `KeyManageController.cpp`：`tryAutoReturnCompletedTicket()`（约 2154~2177 行）

## P10：人工传票失败后仍可能被静默转为自动补发（高）

### 现象

用户在工作台手工点击一次“钥匙传票”，如果这次人工传票失败，理论上应该是：

- 失败就结束
- 由用户自己决定是否再次点击重试

但当前实现里，人工失败后仍可能被系统自己改成 `auto-pending`，后续在钥匙状态恢复后自动再次发送。

### 根因

当前至少有两条代码路径会把人工失败重新塞回自动传票队列：

#### 路径一：底层命令在途时的失败转 auto-pending

`TicketTransferFailed` 分支（`KeyManageController.cpp` 约 1236~1250 行）里，只要失败原因包含“当前有命令在途”，就会：

1. 把该票 `updateTransferState(..., "auto-pending")`
2. 清掉活动传票上下文
3. `scheduleAutoTransferRetry()`

这条路径不区分最初是人工触发还是自动触发。

#### 路径二：钥匙空任务时的失败票自动回队

`recoverAutoTransferWhenKeyEmpty()`（`KeyManageController.cpp` 约 2002~2046 行）里，只要后续 `Q_TASK` 发现钥匙任务列表为空，就会把：

- `sending`
- `failed`

状态的票重新改成 `auto-pending`，然后再触发 `tryAutoTransferPendingTicket()`。

### 业务影响

- 一次人工点击失败后，系统可能不会真正停下
- 用户以为“失败已经结束”，系统却可能在未来自己再发一次
- 用户侧结果与设备真实动作分叉，影响现场判断

### 涉及代码

- `KeyManageController.cpp`：`TicketTransferFailed` 中 inflightBusy 分支（约 1236~1250 行）
- `KeyManageController.cpp`：`recoverAutoTransferWhenKeyEmpty()`（约 2002~2046 行）

## P11：HTTP传票安全超时只清了表层锁，未真正清掉卡住的传票上下文（高）

### 现象

工作台传票请求如果卡住 60 秒，日志会显示：

- “HTTP transfer transaction safety timeout fired”
- “强制释放”

但后续再次点击传票时，程序仍可能继续提示：

- 当前已有其他传票任务正在执行

看起来像“已经超时释放”，实际上仍然被旧上下文挡住。

### 根因

HTTP 事务安全超时回调（`KeyManageController.cpp` 约 123~135 行）调用的是：

- `clearHttpTransferTransaction(false)`

而 `clearHttpTransferTransaction()`（约 2411~2435 行）在传入 `false` 时，只会清：

- `m_httpTransferTransactionActive`
- `m_httpTransferTransactionTaskId`
- `m_httpTransferTransactionSessionId`

不会清：

- `m_activeTransferTaskId`
- `m_activeTransferSessionId`
- `m_pendingTransferHandshakeTaskId`
- `m_pendingTransferHandshakeJsonBytes`

而后续 `tryStartTicketTransfer()`（约 2454~2469 行）仍然会因为 `m_activeTransferTaskId` 非空而拒绝新传票。

### 业务影响

- 超时日志和真实业务状态不一致
- 看起来像“已解锁”，实际上还是被旧任务卡住
- 现场可能只能重启程序恢复

### 涉及代码

- `KeyManageController.cpp`：HTTP 事务安全超时回调（约 123~135 行）
- `KeyManageController.cpp`：`clearHttpTransferTransaction()`（约 2411~2435 行）
- `KeyManageController.cpp`：`tryStartTicketTransfer()`（约 2454~2469 行）
