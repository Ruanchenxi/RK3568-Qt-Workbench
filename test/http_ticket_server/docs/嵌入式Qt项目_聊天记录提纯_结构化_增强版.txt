【模块 1：项目基础信息】
- 项目名称：http_ticket_server（复刻原产品：工作台 Web 传票 JSON → 本地 HTTP → 操作票二进制编码 → 串口票据帧 → Golden 对比/回放）
- 核心目标：
  - 在 Windows 开发环境先完成闭环：PowerShell/工作台 POST JSON → 本地 8888 接口接收 → 记录/落盘 → 编码为票据帧 HEX → Golden 对比 PASS
  - 后续部署到 RK3588：与原产品同机运行，接入工作台点击“钥匙传票”触发真实下发
- 技术栈：Qt 5.15.x（QtWebEngine 嵌入工作台；QtNetwork/QTcpServer 本地 HTTP；C++；GBK 编码处理；CRC16 算法）
- 核心功能（完成状态，按推进阶段）：
  - Step1：独立 HTTP Server 监听 127.0.0.1:8888，支持 POST 接收 JSON 并返回 {success,msg} → 已完成
  - Step2：HTTP 服务端报文日志（时间戳/Origin/User-Agent/pretty JSON/落盘 logs/http_ticket_*.json）→ 已完成
  - Step3：CORS + OPTIONS 预检兼容工作台 WebEngine；兼容 body 可能带 data 外壳 → 已完成
  - Step4.1：生成“传票串口帧”HEX（不写串口），输出 Len/CRC/分帧信息 → 已完成
  - Stage5.1+5.2：JSON→payload 编码（version/taskId/stepNum/time/stepTable/GBK 字符串区 + 偏移回填）→ 已完成
  - Stage5.4：Golden 回放/对比 Runner（--replay / --golden / --out），可 PASS 到原产品抓包帧 → 已完成
  - Replay / Diff Tab：支持“使用最新收到的 JSON”与“自动跟随最新 HTTP JSON” → 已完成
  - Golden 工具 Tab：支持多步样本解析展示 → 已完成
  - 真机 ACK：串口助手发送帧后收到设备 ACK 回包 → 已完成（仅确认链路）
- 硬性约束：
  - 字符串区：GBK 编码 + 00 结尾；VNC 复制中文可能出现 ????，以帧中 GBK 字节为准
  - 偏移字段：4 字节小端回填（taskNameOff/displayOff 等）
  - 时间字段：BCD 编码；当源时间仅到分钟时，末两字节使用 FF FF（以实测 PASS 版本为准）
  - Golden 规则：逐字节完全一致（含 CRC）才算 PASS

【模块 2：串口协议核心规则】
- 基础通信参数：115200，8N1，无校验，无流控
- 外层帧结构（传票/操作票命令族）：
  - SOF(2)=7E 6C
  - KeyId(1)
  - FrameNo(1)
  - Station(1)
  - Device(1)
  - Len(2,LE)：长度=Cmd(1)+Seq(2)+PayloadLen
  - Cmd(1)：0x03 单帧/最后帧；0x83 表示还有后续帧（bit7=1）
  - Seq(2,LE)：数据帧号，从 0 开始每帧+1
  - Payload(n)
  - CRC(2,BE)：对 [Cmd][Seq][Payload] 计算；帧尾写 CRCHi CRCLo；初值 0
- 内层 payload（操作票文件数据）当前已验证布局：
  - version(2)：01 11
  - reserved(2)：00 00
  - taskId(raw)(16)：当前实测为 uint64(LE,8B) + 00×8
  - stationId(1)
  - ticketAttr(1)：00=普通票
  - stepNum(2,LE)
  - taskNameOff(4,LE)：回填 taskName 字符串起始偏移
  - createTime(8,BCD)：缺秒时写 FF FF
  - planTime(8,BCD)：同上
  - auth(48)：当前实现 FF×48（TODO）
  - reserved(剩余固定区)：按长度填充 FF
  - stepTableStart = p[160]
  - 每步固定 8B：innerCode(2,LE) + lockerOp(1) + state(1) + displayOff(4,LE 回填)
  - stringAreaStart = 160 + 8 * stepNum
  - 字符串区：stepDetail[0..N-1](GBK+00) + taskName(GBK+00)
- Golden 验证规则（Stage5.4）：
  - 输入：sample_http.json（含中文） + sample_golden.hex（原产品抓包帧）
  - 输出：生成帧 hex 并与 golden 逐字节 diff
  - PASS：所有字节一致（含 CRC）
  - FAIL：只输出首差异 offset + 字段标签 + 差异片段（首差异 + 关键字段标注）

【模块 5：关键问题与解决方案】
- 问题：PowerShell/工作台请求无法连接 localhost:8888
  → 解决：先确认 netstat :8888 是否 LISTENING；复刻程序必须实际启动 HTTP Server 才能测；先做独立小工具验证后再移植
- 问题：工作台前端 toast 需要 success/msg 字段
  → 解决：接口必须返回 HTTP 200 + JSON {"success":true/false,"msg":"..."}；不能只回空 200
- 问题：浏览器跨域导致请求被拦（嵌入 QtWebEngine 也会触发）
  → 解决：实现 CORS 头 + OPTIONS 预检；Allow-Origin/Methods/Headers
- 问题：VNC/复制中文变 ???? 导致无法作为黄金样本
  → 解决：以串口帧中 GBK 字节为准；通过 golden.hex 反解 GBK 得到中文并生成 replay.json/sample_http.json
- 问题：黄金样本 planTime 不一致导致无法对齐原产品 CRC
  → 解决：以原产品帧中的 BCD 时间为准（例如 09:29 对应 planTime=202603050929），样本 JSON 必须匹配
- 问题：分帧/Len/统计不一致导致难以对齐
  → 解决：Len 统一按 Cmd+Seq+payloadChunk；日志中 payloadChunkLen 必须与 Len 一致；原产品 286B 多见单帧，优先单帧对齐

【模块 7：核心共识】
- 主链路共识：本地 HTTP 接口接收工作台 JSON 是主入口；串口下发是下游；Golden 对齐是验收标准
- 编码/验证优先级共识：先做到逐字节对齐 golden（含 CRC）→ 再真机 ACK 闭环 → 再扩展多步/扩展字段/自动串口
- JSON 解析共识：有 data 就取 data；无 data 就用根对象；外层 code/msg/success 等不进入 payload
- 输出/可复现共识：所有关键环节必须可落盘（logs/）与可复制（SEN_HEX），避免“口头对齐”
- 字符串与时间共识：中文以 GBK 字节对齐原产品；时间以帧内 BCD 实测为准（不以系统当前时间推导）
